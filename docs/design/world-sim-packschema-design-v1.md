# 世界パックスキーマ設計書 v1 ― エンジン/パック境界の型定義と用語集

Owner: G9 DDD設計者／ラウンド内レビュー: G10 アーキテクト・G6 世界考証・G14 検証工学（帽子を掛け替えて実施）／批評: G17
本書は**境界の型（器）**を定義する。データの中身は水滸伝パック仕様書v1が正であり、本書は一行も追加しない。
実装は含まないが、M0実装ラウンドがTypeScript型へ書き起こす直接の元本となることを想定して書く。

## 0. 位置づけと原則

### 0.1 位置づけ

M0ゲート「世界パックスキーマ凍結」の本体。構造設計書§2.2の概念スキーマを、実装発注可能な
型定義へ具体化する。分離は後から入れられない（固有名詞は放置すると全層に根を張る——構造§2.2）
ため、**スキーマは実装初日から存在する**。

### 0.2 原則

1. **if文はエンジン、名詞はパック**（憲法④）。本書はこの判別規則の型表現である。
2. **パックは純データである。** パックはコード・式・条件分岐を一切含まない宣言的データのみで構成する。パックの表現力が足りない場合、パック側に規則を書く道は存在せず、**エンジン側プリミティブの追加をCRで要求する**（パック仕様書§15第2回の運用の型による強制）。パックにロジックを許すことは台本禁止則の裏口を開けることである——本原則は台本禁止則の子として導出する。
3. **境界を跨ぐ参照はIDのみ。** エンジンはパックをIDと型で解釈し、具体パックの存在を知らない。パック内の相互参照もIDのみ（人物→所属制度、技→プリミティブ等）。
4. **禁止は構造で強制する。** 憲法違反（初期怨恨の仕込み等）は「入力しないでください」ではなく、**該当フィールドがスキーマに存在しない**ことで封じる（§4）。

## 2. スキーマ定義

記法：概念型定義（TypeScript風の擬似記法）。`enum` と明記した列挙はエンジン側の凍結語彙。

### 2.1 ルートと互換性

```
WorldPack = {
  meta: { packId, packName, version, engineSchemaVersion, 出典注記? }
  geography, agents, institutions, skills, causes, agitation,
  vocabulary, eraParams, evaluationOverrides
}
```

- `engineSchemaVersion`：パックが準拠するスキーマ版。エンジンは同一major・以下minorを受理。
- スキーマ自体の変更は本書の改訂（凍結後はCR経由）であり、凍結API扱い。

### 2.2 地理 geography

```
NodeDef = { id, nodeType: enum{府城,県城,村鎮,山寨適地,水泊,通過点,関所},
            富, 人口, 民心, 治安, 執行力, 防御地形値,
            encounterWeight,  // 遭遇地形係数（パック§9。劇的バイアスとは独立）
            regionTag }       // 地方タグ（攪拌の対象指定用）
EdgeDef = { id, from: NodeId, to: NodeId, kind: enum{幹線,支線,水路}, 距離 }
```

- 「渡し場・宿場・酒店」は`通過点`型＋encounterWeightで表現する（パック§1.1の遭遇率ボーナス最大）。
- 護送が幹線を通る規則は制度側（§2.4 移送規則）が参照する。地理は事実のみを持ち、規則を持たない。

### 2.3 人物 agents（三層供給——パック§2.1）

```
agents = { explicit: AgentDef[],        // S/A層（手書き）
           archetypes: ArchetypeDef[],  // B層・世代交代の生成原型
           generation: { 初期B層人数帯, 世代交代の登場規則パラメータ } }

AgentDef = { id, 姓名, 渾名?, 出自, 開幕ポスト?: PostRef, 開幕所在: NodeId,
             素質: {武勇,知略,統率,魅力,技芸},
             価値観: {利他,忠誠,野心,獲得,攻撃性,慎重,体面,愛着},   // エンジン抽象軸名
             初期関係: InitialRelation[], 初期資産, 初期名声: ScopedFame[],
             出典注記 }   // S/A層は必須（パック§14-1：解釈をデータ改訂で変更可能に保つ）

InitialRelation = { 対象: AgentId, 種別: enum{血縁,婚姻,師弟,同僚}, 軸初期値 }
ArchetypeDef = { id, 原型名参照: VocabRef, 価値観分布: {軸→(平均,分散)}, 素質傾向, 出自語彙: VocabRef }
ScopedFame  = { 観衆: CommunityId, 値 }   // CR-3の器。M3実装まで観衆は単一でも可
```

- **`初期関係.種別`は4種のホワイトリスト列挙である。** 怨恨・恩義・結義を初期関係に書く手段はスキーマ上存在しない——運命の白紙原則（パック§0-2）の構造的強制。
- 価値観軸名はエンジン語彙（抽象名）。「義侠」等の文化的表示名は vocabulary が与える（構造§1.2の再配置を型で固定）。

### 2.4 制度 institutions（構造§2.3の器）

```
InstitutionDef = { id, 名称: VocabRef, 管轄: NodeId[] | '全土',
                   規則群: RuleParam[], ポスト群: PostDef[],
                   腐敗度, 正統性 }
RuleParam = { 規則型: enum{課税,登用,処罰,叙勲,治安,移送}, パラメータ組 }
PostDef   = { id, 名称: VocabRef, 定員, 俸給, 付与行動タグ: enum[], 面子値 }
```

- 規則型は**エンジンが解釈する6種の凍結列挙**。処罰規則のパラメータに刺配（烙印付与＝CR-1の器）・護送要否を含み、移送規則が幹線経路制約を持つ（パック§3.2の護送を型で受ける）。
- 「朝廷」は`管轄:'全土'`の一インスタンスに過ぎない。特別扱いのフィールドは存在しない。

### 2.5 技 skills（プリミティブ合成——構造§2.4）

```
SkillDef = { id, 表示名: VocabRef,
             合成: { プリミティブ: enum{投射,面制圧,延焼汚染,地形改変,恐慌士気,目標上書き,隠密偽装,移動強制,近接}, パラメータ }[],
             目撃タグ?: VocabRef,      // 例:「超常」（妖術の評価タグ）
             保有条件: { 素質下限?, 出自タグ?, 稀少度 } }
```

### 2.6 大義 causes（構造§4.2の器）

```
CauseTemplate = { id, 名称: VocabRef, 価値観プロファイル: {軸→重み},
                  敵対対象型: enum{制度,勢力,状態,なし}, 正統性基盤: enum{江湖名声,官途名声,血統,自衛} }
```

### 2.7 攪拌 agitation（構造§1.2の再配置先）

```
AgitationEntry = { id, 型: enum{災害,収奪,政変,外圧,疫病,大赦},
                   実体名: VocabRef, 対象: regionTag | '全土' | '抽選',
                   頻度帯, 強度パラメータ }
```

- 型6種はエンジン語彙（`大赦`はパック§7の「烙印者の一部赦免」を汎用化した戻し型）。遼・金・黄河はすべて実体名＝語彙参照。

### 2.8 語彙 vocabulary

```
vocabulary = { 人名: { 姓プール, 名の型規則 },
               渾名: { 型分類: enum{獣身体,天象霊獣,古人比擬,職能誇張,性向}, 付与キー→語彙表 },
               題テンプレ: Template[], 台詞: { (場面,性格,関係)キー → 台詞群 },
               史官文体: StyleParam, 表示名map: { エンジン語彙キー → 文化的表示名 },
               観衆: CommunityDef[] }   // CR-3: 官界/江湖/民間はここで定義
Template = プレースホルダ置換のみの文字列（例: "{地名}の{戦い|乱|変}"）
```

- **テンプレは置換のみ。条件分岐・演算は持たない**（純データ原則）。条件が必要な表現はエンジン側キー（場面・性格・関係）の細分化をCRで要求するか、LLM装飾層（G15）の責務とする——テンプレ層は決定論の床であり、文芸の天井はLLM層が持つ（軸B、パック§8のG5/G15合意）。

### 2.9 時代パラメータ eraParams

寿命分布・移動速度（徒歩/騎馬/舟）・貨幣単位・宴会頻度。全て数値パラメータのみ（パック§9）。

### 2.10 評価重み上書き evaluationOverrides（パック§4の器）

```
evaluationOverrides = { 恒常: { 評価場面キー: enum → 倍率 },
                        一時状態: { 状態キー: enum{飲酒,…} → { 軸→倍率, 持続 } } }  // CR-2の器
```

- 評価場面キー（不義の目撃・公衆侮辱・結義紐帯・恩義未清算・制度不信…）はエンジン側enumであり、
  **文化＝同じ軸の発火のしかたの偏り**（構造§7-1）はこの表の倍率としてのみ表現できる。

## 4. 検証規則（スキーマ適合の機械検査——G14監査群と対）

パックのロード時および CI で実行する（M0実装対象）：

1. **構造検証**：必須フィールド充足、全ID参照の解決（存在しないノード・制度・語彙への参照ゼロ）、ID重複ゼロ、循環参照ゼロ。
2. **憲法検証**：
   - 初期関係の種別がホワイトリスト4種のみ（白紙原則。スキーマ構造＋バリデーション二重化）
   - 特定ノード・特定人物への特別フラグの不存在（偏在の禁止——スキーマにフィールド自体が無いことの確認を含む）
   - S/A層人物の出典注記の存在（パック§14-1）
3. **境界検証**：固有名詞監査（検出器仕様書§5.1）の禁止リストを vocabulary＋全実体名から自動生成できること。生成不能な名詞フィールドの存在はスキーマ違反。
4. **数値検証**：全数値が定義域内（軸0〜100等）。分布パラメータの正値性。

## 5. 想定される問題点

1. **評価場面キーenumの網羅性**——パック§4の6場面で百年分の文化表現に足りるか。不足時の拡張はエンジンCRになり足が重い。→ M3のデータ入力（最小舞台）で不足を実測し、必要ならまとめて1件のCRとする。
2. **生成規則のデータ表現の限界**——渾名付与キーや世代交代規則は「規則らしさ」が強く、純データ制約と緊張する。→ 本書は「キー→表」の形（判断はエンジン、対応表はパック）まで許し、それを超える表現はエンジンプリミティブ化。境界事例はG9/G10レビューで個別裁定し用語集に前例を残す。
3. **B層生成とシード**——原型からの人物生成はエンジンのシード乱数（世界生成ストリーム）で行う。パックは分布のみを持ち、乱数を持たない（決定論はエンジンの責務）。

## 6. 批判章（G17）

1. **純データ制約は本当に守れるのか。** 台詞テーブル・題テンプレは既に「ミニ言語」の入口にいる。プレースホルダに書式指定が生え、次に条件が生え、三年後にパックがスクリプト言語を持つ——これがこの種の設計の典型的な死に方である。→ 応答：防波堤は「テンプレは置換のみ」の一行だけでは弱い。**M0実装でテンプレパーサを意図的に貧しく作り（置換以外を構文エラーにする）、拡張要求はすべてCR経由で本書の改訂を要する**ことを凍結条件に含める。構造で強制せよ、の自己適用。
2. **スキーマの早期凍結はM1〜M3の実装で必ず綻びる。** 型は書いてみるまで穴が見えない。→ 応答：正しい。凍結対象を「構造（節構成・原則・enum群・禁止事項）」とし、フィールドの追加はminor改訂（CR不要、G9レビューのみ）、フィールドの変更・削除とenum変更はCR必須、と改訂規約を二段化する。硬直と無秩序の間に段差を作る。
3. **用語集の権力集中。** G9が正典を握ると、他帽子の概念がG9の言語感覚に翻訳されて歪む。→ 応答：登録の拒否権はG9に置くが、**定義文の Owner は概念の生みの親の帽子**とする（例：検出器の定義はG14の文言を無修正で収録した）。G9は整合性のみを審査し、意味には触れない。

## 7. 自己レビュー（3回）

**第1回（検出器仕様との整合＝発注の合格条件）**：G14の用語4種・DetectionInstanceを用語集へ収録して
整合を確認した。ただし検出器・本書ともに未定義のイベント型名を参照している。→ 改善：イベント型名は
すべて仮称であり、G11イベントスキーマ設計ラウンドで正式化される旨を§2に追記し、両書が同じ
プレースホルダ規約（仮称に`（仮）`を付さず、G11ラウンドでの一括置換を凍結条件とする）に従うことを明記した。

**第2回（CR-2の置き場所）**：当初、飲酒修飾を vocabulary 側に置いていたが、CR-2は「エンジン気分機構の
拡張」でありパックが持つのは倍率と持続のみ。→ 改善：evaluationOverrides.一時状態へ再配置（§2.10）。
再設計ではなく再配置で直る違反を、自書の中でも見逃さないこと。

**第3回（パック仕様書との逐条突合）**：パック§1〜§9の全データ項目が載るかを確認した結果、
護送規則（§3.2）を受ける器が制度の規則型に無かった。→ 改善：規則型enumに`移送`を追加（§2.4）。
また§7攪拌の「大赦」が災害/収奪/政変/外圧/疫病の5型に収まらないことを発見し、型enumへ`大赦`を
追加した。**スキーマの穴は突合でしか見つからない**——この突合手順自体をM0実装の検収（§4-1の
テストケース）に含める。

## 8. 二重検問への回答文

**問1（プレイヤーを削除しても面白いか）**：本スキーマにはプレイヤーに関するフィールドが一つも
存在しない（フォロー・しおり等のメタデータは observatory 側の別ストアであり、パックにもエンジン
状態にも属さない——観測§6.1）。スキーマが記述するのは世界の初期条件と文化的偏りのみであり、
その世界は定義上、観客なしで百年回る。YES。

**問2（再現か、生成か）**：本スキーマは筋書きを書く場所を構造的に持たない。初期関係は開幕前から
存在した4種のみに型で制限され、初期怨恨・運命フラグ・特定ノード優遇のフィールドは存在しない。
パックに書けるのは「舞台と偏り」だけであり、そこから何が起きるかを書く手段がない——器の形そのもの
が「生成」の側に立っている。YES。
