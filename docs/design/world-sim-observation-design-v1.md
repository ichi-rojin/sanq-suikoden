# 観測系設計書 v1 ― 面白さを発見可能にする第二のエンジン

本書は `world-sim-design-v1.md` の§10〜11を全面拡張し、**観測系をゲームデザインの中核**として再設計する。実装は含まない。

---

## 0. 前提となる認識の修正

v1設計書§17-1で指摘した通り、本作の急所は「シミュレーションは深いが、面白さが埋もれる」ことである。
この問題への回答として、本作を**二つのエンジンの対**として再定義する。

- **世界エンジン**：面白さを*生成*する（v1設計書の主題）
- **観測エンジン**：面白さを*発見・理解・振り返り可能*にする（本書の主題）

観測エンジンは世界エンジンのビューアではない。**対等な製品コンポーネント**である。
世界エンジンがどれほど良い歴史を生成しても、観測エンジンが凡庸ならプレイヤーには存在しないのと同じである。逆に観測エンジンが優れていれば、凡庸な出来事すら「発見の喜び」に変換できる。

### 0.1 プレイヤー像の再定義

Phase1のプレイヤーは「観客」ではない。**史家（歴史家）**である。

- 観客は流れてくるものを見る。受動的で、密度が薄い時間は苦痛になる。
- 史家は世界を**調べる**。気づき、追跡し、因果を解き、年代記を編む。能動的な営みである。

Phase1の「プレイ」とは、**世界を素材とした調査・読解・編纂行為**である。これにより「操作できないゲーム」は「操作対象が世界の理解である ゲーム」になる。

### 0.2 観測不干渉原則（絶対規則）

観測系は世界へ**1bitも書き込まない**。フォロー・検索・注目は世界の進行に一切影響しない。
理由：v1§1.4（観察者不在原則）の帰結。「注目した武将が活躍しやすくなる」類の設計は、発見の喜び（世界が勝手にそうなった）を根本から破壊する。遭遇の劇的バイアス（v1§18-1）は世界エンジン内部の環境規則であり、プレイヤーの注目とは無関係であることをここで明確化する。

### 0.3 注意の漏斗

100年の世界は数百万イベントを生む。プレイヤーの注意は1セッション数時間しかない。
観測系の仕事は、この比率を段階的な漏斗で埋めることである。

```
数百万イベント
 → 数千（インデックス化された検索可能単位）
 → 数百（編纂された物語・年報）
 → 数十（プレイヤーの注目領域に交差するもの）
 → 数件（今この瞬間、画面に出すべきもの）
```

各段が独立したシステムを持つ。下流ほど編集的判断が濃くなる。

## 1. 中核体験ループ

観測系の全機能は、この5拍子ループのどこかに奉仕する。奉仕しない機能は作らない。

```
① 気づく（Discovery）   … 号外・定期報・異変・注目通知
② 追う（Tracking）      … フォロー・監視クエリ・伏線リスト
③ 分かる（Understanding）… なぜ？ボタン・人物帳・相関図・地図レイヤ
④ 遡る（Retrospection） … 歴史書・年代記・リプレイ・検索
⑤ 編む（Curation）      … しおり・私撰史書・時代命名・注釈
        ⑤が新たな①を生む（編纂中の発見）
```

設計検証の問い：**「この画面はループのどの拍に効くか？次の拍への導線があるか？」**
例：号外（①）には必ず「なぜ？」（③）と「フォロー」（②）の導線を付ける。行き止まりの画面を禁止する。

## 2. 発見系（Discovery）

### 2.1 号外（速報）

顕著度が閾値を超えたイベント（建国・滅亡・大戦・著名武将の死・裏切り・招安）を即時に画面へ差し込む。
- 形式：一行見出し＋関係者の顔＋発生地への地図ジャンプ。**見出しは史官の題生成器（v1§10.3）を流用**し、事務的なログ文ではなく「大名府、陥落」「豹子頭、官府に叛す」の文体で出す。
- 頻度制御：号外は1年あたり上限数を持ち、超過分は定期報へ降格。速報の希少性を守る。

### 2.2 定期報（旬報・月報・年報）

史官が定期的に書く「新聞」。ダイジェストの主力。
- **旬報**：低速再生時の伴走。地域ごとの小見出し数件。
- **月報**：勢力動向・人事・遭遇の要約。
- **年報**：その年の総括。主要物語への入口、死亡者一覧、勢力図の変化、**「今年の十大事件」**（顕著度ランキング）。
- 高速再生（100倍速・年スキップ）では旬報・月報を自動スキップし年報のみ蓄積する。**再生速度と情報粒度の連動**が、眺めるゲームのテンポ設計の核心。

### 2.3 注目（フォロー）と監視クエリ

- **フォロー**：武将・勢力・地域・党・関係（例：林冲⇔高俅の対）を登録。関連イベントが通知対象になる。
- **監視クエリ**：検索条件（§5）をそのまま保存して通知化する。「梁山泊周辺での戦闘」「未清算怨恨の清算イベント」「結義の成立」など。**検索と通知を同一機構にする**ことで、プレイヤーの興味の言語化がそのまま観測装置になる。
- フォロー対象の通知には優先度が付き、号外と同格に扱われる。

### 2.4 異変フィード（メトリクスの物語化）

世界健全性メトリクス（v1§16-4）は開発用に留めず、プレイヤー向けに翻訳して出す。
「北方で戦争頻度が急増している」「放浪者の数が過去最多」「山東の民心が崩れつつある」——**統計の変化を予兆として読ませる**。異変フィードは「これから何かが起きる」という前方向の期待を作る唯一の装置であり、事後報告ばかりの観測系に未来時制を導入する。

### 2.5 伏線リスト（未完の物語）

v1§18-1-2の「未完の物語」追跡をプレイヤーへ開放する。
- 「転」まで進んで「結」の無い因果塊、未清算の重い台帳、生き別れの義兄弟、滅亡勢力の残党——を**「くすぶる火種」一覧**として表示。
- 各項目はフォロー可能。「この怨恨はいつ爆発するのか」という**待つ楽しみ**を明示的なゲーム要素へ昇格させる。
- これは観測不干渉原則に反しない。世界側の遭遇バイアスは台帳から独立に働いており、リストは既存情報の表示に過ぎない。

## 3. 理解系（Understanding）

### 3.1 「なぜ？」ボタン（因果エクスプローラ）――観測系の看板機能

あらゆるイベント・物語・関係・台帳エントリに「なぜ？」が付く。
押すと causes[] を遡り、**因果の鎖を物語文体で一列に**表示する：

```
Cが離反した。
 ← 宴会でAと口論した（3年前）
 ← Aへの怨恨を抱えていた（8年前に記帳）
 ← 飛雲渡の戦いで、Aの放った矢がCに当たった（8年前）
```

- 各行はさらに展開・ジャンプ可能（当該イベント、当該物語、当該人物帳へ）。
- 分岐（原因が複数）は折り畳みで表現。既定では**最重量経路**（weight最大の因果鎖）のみを一本道で見せ、初見の認知負荷を抑える。
- 逆方向「それで？」（影響先の探索）も対で用意する。原因を遡る快楽と、結果を追う快楽は別物である。
- v1§16-4の考古学CLIと同一のクエリ基盤を使う。**デバッグツールと看板機能が同じ骨格**であることは、因果保存原則の投資対効果の証明でもある。

### 3.2 人物帳

一人の武将を理解するための綴じ込み。タブ構成：

1. **年表**：人生の全記録。物語参加項目はリンク（v1の林冲要件）。無所属期間・所属期間を帯で色分けし、放浪→入山→独立の人生の形を一目で見せる。
2. **台帳**：恩義と怨恨の一覧。各エントリに原因イベントと清算状況。**「この男は誰に借りがあり、誰を恨んでいるか」が人物理解の最短経路**である。
3. **縁**：エゴネットワーク（その人物中心の相関図）。義兄弟・仇敵・旧主・恋人ラベル付き。
4. **評判**：名声・悪名・渾名の変遷。噂システム導入後は「世間が知る彼」と「実際の彼」の差分も。
5. **視点モード**：世界のイベント流を**この人物が知覚したものだけ**にフィルタして読む。憑依観察（v1§17-5）の読書版であり、Phase2への布石。

### 3.3 相関図（時間軸付き）

- 全体グラフは描かない（500人の毛玉は理解不能）。**フォーカス人物からの2ホップ**か、**勢力・党単位**で切る。
- **時間スライダ**：任意の年の関係状態を表示。「30年目」と「60年目」の**差分モード**（新たに結ばれた縁＝緑、切れた縁＝赤、反転した縁＝紫）で、時代の変化を関係の変化として読ませる。
- エッジをクリックすると、その関係の歴史（出会い→結義→決裂の年表）が開く。**関係それ自体が年表を持つ**。

### 3.4 地図レイヤ

地図は現在状態の表示に加え、理解用レイヤを重ねられる：
- 勢力圏の変遷（年スライダで国境が動く）
- 事件ヒートマップ（期間指定：どこで歴史が濃かったか）
- 個人の足跡（人物帳から起動：放浪の軌跡を線で描く）
- 因果の地理表示（「なぜ？」の鎖を地図上の矢印列として描く——誤射の地から離反の地、開戦の地へ）

### 3.5 勢力志

勢力ごとの興亡記。結成動機（原因イベント）、歴代当主、構成員の出入り、版図変遷、内紛史、滅亡（または現況）、残党の行方。滅亡勢力のページは消えない。**死んだ勢力が読める**ことが、興亡の世界の厚みになる。

## 4. 振り返り系（Retrospection）

### 4.1 歴史書リーダー（物語パッケージの読書UI）

編纂済み物語（v1§10）の読書体験を規定する。

- **書架**：物語を年代・地域・人物・型（戦記/離反/結義/救出/興亡/悲恋…）の棚で並べる。品質スコア上位は「名編」として装丁を変える。
- **本文**：出来事列を章立てで表示。人名・地名・勢力名は全リンク。保存済み台詞（v1§10.4）は会話体で組む。
- **相互参照**：本文中の各イベントに「なぜ？」「それで？」「この時、世界では」（同時期の他所の出来事）を付ける。一冊の物語が世界全体への入口になる。
- **続篇リンク**：物語の結末イベントが別の物語の原因になっている場合、「この物語は『◯◯』へ続く」を自動表示。因果グラフ上で物語同士が連結している事実を、読書導線に変換する。
- **未読管理**：既読・積読・読了マーク。書架に「あなたがまだ読んでいない名編」を出す。

### 4.2 年代記タイムライン

- 100年を一本の帯で俯瞰。物語・号外・勢力の存続期間を段組み表示。
- ズームで年→月→日へ。任意時点をクリックすると、その時点の世界状態（地図・勢力図・生存者）を**その時点の知識だけで**表示するリプレイビューへ入る。
- **時代区分の自動検出**：戦争頻度・勢力数の変化点から「乱世」「小康」「大戦期」を検出し、帯に色分け表示。歴史に章立てを与える。

### 4.3 ネタバレ管理（二つの読み方）

観測系は二つの消費様式を明確にサポートする：

- **観戦モード**（ライブ）：世界の現在に伴走する。未来は存在しないので問題なし。
- **読史モード**（事後）：完走済みの100年を後から読む。ここでネタバレ問題が発生する。年代記の任意時点へ跳んだとき、既定では**その時点より未来の情報を隠す**（人物帳の年表は当該時点まで、勢力志は現況を伏せる、「それで？」は無効化）。
- 全知トグルで解除可能。「知らずに読む」と「知って読み返す」は別の快楽であり、両方を提供する。特に読み返し時、「なぜ？」で辿った因果を全知状態で再訪する体験（あの誤射がここに繋がっていたのか）は本作の醍醐味なので、**読了後の全知再読を促す導線**（読了時に「原因年へ跳ぶ」ボタン）を置く。

### 4.4 検索・フィルタ

- **ファセット検索**：人物 × 勢力 × 地域 × イベント型 × 期間 × 顕著度帯 の組合せ絞り込み。「40〜50年目、山東、離反イベント」が3タップで出る。
- **関係検索**：「AとBが両方関与したイベント」「義兄弟が敵対した例」「未清算のまま当事者が死んだ怨恨」——**台帳・関係を検索対象に含める**ことが本作固有の検索価値。
- **全文検索**：物語本文・台詞・題名。
- **保存検索 = 監視クエリ**（§2.3と同一機構）。
- 検索結果は常に年代記・地図へ射影可能（結果を帯の上のマーカー列として見る）。

## 5. 編纂系（Curation）――プレイヤーが史家になる

観測系の最終拍。読むだけの受動性を、編む能動性へ転化する。**世界には一切書き込まない**（プレイヤーのメタデータ層に保存）。

- **しおり・注釈**：任意のイベント・物語・人物に私的メモを残す。
- **私撰史書**：物語・イベント・注釈を束ねて自分の歴史書を編む。章立て・表題はプレイヤーが付ける。史官の正史に対する「私の水滸伝」。
- **時代命名**：年代記の区間に元号・時代名を与える（「三山鼎立の時代」）。以後、観測系全体の日付表示に反映される（表示のみ）。
- **エクスポート**：私撰史書・人物帳・年代記を文書として書き出す。**世界の外へ持ち出せる歴史**は、このジャンルの布教装置でもある（シード共有と組で「同じ世界を読んだ者同士」の語らいが成立する）。

編纂系はPhase1の「勝利条件なきゲーム」に**作品という成果物**を与える。プレイヤーは世界に勝てないが、世界について最良の本を書くことはできる。

## 6. アーキテクチャへの反映

### 6.1 書込モデルと読取モデルの分離（CQRS構造）

世界エンジンは書くだけ、観測エンジンは読むだけ。両者はイベントログとスナップショットのみで接続する。

```
@world/core ──(EventLog / Snapshots)──▶ @world/observatory（読取モデル）
                                             ├ インデクサ（人物/地域/勢力/型/期間/台帳）
                                             ├ 因果クエリ（なぜ？/それで？/最重量経路）
                                             ├ 集計器（メトリクス→異変フィード）
                                             └ 検索器（ファセット/全文）
@world/chronicle ─(StoryShelf)─▶ observatory（書架・年報の供給元）
@world/viewer ──▶ observatoryのクエリAPIのみを叩く（coreへ直接触れない）
プレイヤーのメタデータ（フォロー/しおり/私撰史書）… observatory側の別ストア。世界セーブと分離
```

- 観測不干渉原則がアーキテクチャで強制される：viewerからcoreへの参照経路が存在しない。
- インデックス構築はtick処理と非同期で良い（読取モデルの鮮度は数tick遅れて構わない唯一の場所）。世界エンジンの決定論・性能予算を観測系が食わない。
- 考古学CLI（v1§16-4）は observatory のクエリAPIをCLIから叩く薄い皮になる。**開発ツールと製品機能の基盤統一**。

### 6.2 インデックス設計（読取モデルの中身）

| インデックス | キー | 主な供給先 |
|---|---|---|
| 人物別イベント | agentId × tick | 年表・視点モード・足跡 |
| 地域別イベント | nodeId × tick | 地図ヒートマップ・地域報 |
| 型別イベント | eventType × tick | ファセット検索・基準シナリオ検出 |
| 因果隣接 | eventId → causes / effects | なぜ？・それで？・続篇リンク |
| 台帳状態 | (from,to) × 清算状況 | 伏線リスト・関係検索 |
| 関係スナップショット | 年 × エッジ差分 | 相関図の時間スライダ・差分モード |
| 物語索引 | 人物/地域/型/品質 | 書架・年報 |
| メトリクス時系列 | 指標 × 年 | 異変フィード・時代区分検出 |

イベントログはappend-onlyなので全インデックスは増分構築可能。セーブ後の再構築（インデックスはセーブに含めず再生成する）方針にすれば、セーブデータはログ＋スナップショットのまま肥大しない。

## 7. 想定される問題点

### 7-1. 通知の洪水と沈黙の両端
フォローが多いと洪水、少ないと沈黙。→ 号外の年間上限（§2.1）、通知の自動束ね（同一因果鎖の連続イベントは1通に畳む）、無フォロー時は年報と異変フィードが既定の背骨になる設計。

### 7-2. 「なぜ？」の因果鎖が長すぎる／自明すぎる
20段の鎖は読めず、「移動した→到着した」は無意味。→ 顕著度の低い中間イベントを鎖表示から自動省略（省略印「…」で展開可能）。最重量経路の既定表示（§3.1）。

### 7-3. 読取モデルのメモリ
100年分の全インデックス常駐は重い。→ 期間セグメント単位の遅延ロード。観測系はtick同期不要なのでディスク常駐＋オンデマンドで成立する。

### 7-4. ネタバレ管理の徹底コスト
未来情報の隠蔽は全画面に波及する横断的関心事。→ observatoryのクエリAPIに**視点時刻（as-of tick）を必須引数**として組み込み、UI側の個別対応ではなくクエリ層で一括強制する。

### 7-5. 編纂系の空振り
私撰史書は刺さる人には深いが、大半のプレイヤーは使わない可能性。→ 編纂系はM5以降の後載せ可能な独立機能とし、Phase1 DoDには含めない（§9参照）。

## 8. 批判的レビュー

### 8-1. 観測系は「ゲーム」になったか
本書はループ（§1）を定義したが、依然として明確な目標・失敗・上達が無い。史家プレイの動機は好奇心のみに依存する。
反論：好奇心駆動の製品（Wikipediaラビットホール、Dwarf Fortress Legends）は存在するが、**導線設計が生死を分ける**。行き止まり画面禁止（§1）と続篇リンク（§4.1）が最重要規律であり、レビュー時は常に「次のクリック先があるか」を検収項目にする。

### 8-2. 情報設計が世界の質の低さを隠せない懸念
観測系がどれほど良くても、鎖の中身が凡庸なら「よくできた索引の付いた退屈な本」になる。観測系は増幅器であり、生成器ではない。世界エンジンの物語品質（v1§18-1）と本書は**両輪であり、片方で他方を代替できない**ことを明記する。

### 8-3. 開発コストの再配分
観測系を対等なエンジンに昇格させた結果、Phase1の工数は実質1.5倍になった。何かを削る必要がある。→ §9の優先度で応答する。

## 9. 自己レビューと改善（3回）

### 自己レビュー第1回：機能過多。Phase1観測系の最小核を定義せよ

全部は作れない。ループの各拍から最小1機能を選び、**観測系MVP**を確定する：

| 拍 | MVP採用 | 後送り |
|---|---|---|
| ①気づく | 号外＋年報 | 旬報/月報、異変フィード、監視クエリ |
| ②追う | フォロー（人物・勢力のみ） | 伏線リスト、関係フォロー |
| ③分かる | **なぜ？ボタン**＋人物帳（年表・台帳） | 相関図時間スライダ、視点モード、地図レイヤ群 |
| ④遡る | 書架＋本文リーダー＋ファセット検索（人物/期間/型） | 年代記リプレイ、ネタバレ管理、全文検索 |
| ⑤編む | しおりのみ | 私撰史書、時代命名、エクスポート |

判断基準：**因果保存原則の直接の子（なぜ？・台帳・年表・書架）を最優先**。これらは世界エンジンの検証ツールを兼ねるため（§6.1）、開発初期から存在すること自体が世界エンジンの品質を上げる。相関図・リプレイ・編纂は世界が面白いことが確認されてからで遅くない。

### 自己レビュー第2回：v1のDoD（マイルストーン）と統合されていない

v1§18-3のM1〜M5に観測系が乗っていない矛盾を解消する。改訂：

- **M1** 世界骨格＋武将AI＋放浪遭遇＋**observatoryの器（イベントインデックス＋因果クエリCLI）**
- **M2** 関係・感情・台帳＋基準シナリオ2種＋**なぜ？（CLI版）で基準シナリオを検収する運用の開始**
- **M3** 勢力興亡＋抽象戦闘＋**人物帳・年表（最初のGUI）**
- **M4** 史官＋物語棚＋**書架リーダー・号外・年報・フォロー**
- **M5** バランス＋**ファセット検索・しおり**＋DoD検収

観測系を後付けの「UI工程」にせず、**各マイルストーンの検収装置として同時に育てる**。これが§8-3のコスト増への実質的な回答でもある（開発に必要な可視化を製品機能として作るため、二重投資にならない）。

### 自己レビュー第3回：DoDに観測体験の受け入れ基準が無い

v1のDoDは世界の生成品質のみを測っていた。観測系の受け入れ基準を追加する：

1. **五分試験**：完走済み100年世界を初見の読者に渡し、5分以内に「面白い出来事」へ自力到達できるか（年報→号外→物語の導線検証）。
2. **なぜ？往復試験**：任意の号外から「なぜ？」を3回辿って起点イベントへ到達し、そこから「それで？」で号外へ戻れるか（因果インデックスの完全性）。
3. **人物再構成試験**：ランダムな武将の人物帳だけを読んで、その人物の人生の要約を読者が口頭で語れるか（年表・台帳の説明力）。
4. **行き止まり監査**：全画面を巡回し、次のリンクが存在しない終端画面がゼロであること。
5. **不干渉監査**：フォロー・検索を大量に行った実行と無操作実行が、同一シードで同一のイベントログを吐くこと（観測不干渉のリグレッションテスト）。
