# 水滸世界シミュレーション ― 歴史生成エンジン設計書 v1

本書は実装を含まない。設計・分析・分解・批判的レビューのみを記す。
判断基準は常に一つ：**「プレイヤーを完全に削除しても、この世界は面白いか？」**

---

## 1. 世界設計思想

### 1.1 三層構造：状態 → 出来事 → 物語

このエンジンの出力は「ゲーム状態」ではない。**歴史**である。
世界を三層で捉える。

- **状態層（State）**：武将・勢力・地形・経済の現在値。シミュレーションの実体。
- **出来事層（Event）**：状態変化の記録。全出来事は**因果リンク（causes[]）**を持つ。
- **物語層（Story）**：出来事の因果グラフから抽出・編纂された歴史書。プレイヤーが読む対象。

多くのゲームは状態層だけを作り、イベントログを副産物として吐く。
本作は逆である。**出来事層こそが本体**であり、状態層は出来事を生むための土壌、物語層は出来事を人間が読める形に翻訳する装置に過ぎない。

### 1.2 因果保存原則（Causality Preservation）

全ての出来事は「なぜ起きたか」への参照を持つ。
矢の誤射 → 怨恨の発生 → 宴会での口論 → 離反 → 十年後の戦争。
この連鎖が成立するのは、各出来事が原因イベントのIDを保持し続けるからである。

因果保存は以下三システムの共通基盤となる：
- 感情システム（誰のせいで何が起きたかを記憶する）
- 波及システム（出来事が次の出来事を誘発する）
- 物語パッケージ（因果グラフの部分グラフ＝物語）

**この原則が本設計の背骨である。** 他の全システムはこの上に乗る。

### 1.3 決定論原則

同一シード・同一バージョンなら同一の百年史が生成される。
- デバッグ可能性：創発バグの再現に必須。
- リプレイ可能性：イベントログ＋シードから世界を再構築できる。
- LLMによる文章装飾は**シミュレーションに影響を与えない読み取り専用の層**に隔離する（§10, §16参照）。

### 1.4 観察者不在原則

世界はプレイヤーの視界・カメラ・操作から一切の影響を受けない。
LOD（詳細度）による計算削減は許されるが、**結果が変わるLODは禁止**。
「見ていない場所で歴史が進んでいた」という発見こそが本作の快楽であるため、見ていない場所こそ真面目に動かす。

### 1.5 武将主人公原則

主人公は世界であり、その世界は武将たちの人生の総和である。
勢力・戦争・経済は武将の欲望の**結果**として発生する。逆ではない。
設計上、勢力AIというトップダウンの意思決定者は存在しない。勢力の方針は当主という一人の武将の欲望と、配下たちの思惑の合成である。

## 2. コアゲームループ

### 2.1 時間階層

| 単位 | 処理内容 |
|---|---|
| 日tick | 武将の行動決定・移動・遭遇・戦闘進行 |
| 旬tick（10日） | 噂の伝播、関係の自然変化、経済の小更新 |
| 月tick | 勢力方針の見直し、徴税、募兵、外交 |
| 年tick | 加齢・死亡判定・収穫・史官による物語編纂・年表更新 |

100年 = 36,500日tick。武将500人 × 1日1回の効用評価は、評価関数を軽量に保てばヘッドレスで数分に収まる規模（§16で検証項目化）。

### 2.2 日tickの処理順序（決定論的固定順）

1. **世界更新**：天候、火災延焼、季節効果
2. **知覚**：各武将が周辺の出来事・人物を知覚（噂含む）
3. **意思決定**：効用AIが行動を選択（時分割で負荷平準化）
4. **行動解決**：移動、会話、取引、戦闘開始など。ここでイベント発行
5. **反応**：発行イベントに対する感情評価・記憶登録・関係更新
6. **連鎖**：反応が新たな行動候補（復讐、報恩、口論）を生成
7. **記録**：イベントログへ追記（append-only）

反応と連鎖が分離されている点が重要。**出来事は同tick内で完結せず、行動候補として翌日以降に持ち越される。** これが「十年後の戦争」を可能にする。

### 2.3 実行モード

- **CLIモード**：`世界シード＋年数` を与えて全速実行。出力はイベントログ＋物語集＋最終状態スナップショット。
- **ビューアモード**：同一エンジンをブラウザで低速再生。PixiJSは状態を購読して描画するのみ。
- **リプレイモード**：保存済みログを任意時点から閲覧。

## 3. エージェント設計

### 3.1 武将の構成要素

```
武将 =
  素質（武勇・知略・統率・魅力・技芸）
  + 価値観ベクトル
  + 欲求（動的）
  + 目標スタック
  + 記憶
  + 関係（他者への有向エッジ群）
  + 身体状態（年齢・健康・負傷）
  + 社会状態（所属・地位・名声・悪名・資産）
```

### 3.2 価値観ベクトル（不変に近い性格）

水滸伝的テーマで軸を選ぶ。各軸0〜100。

- **義侠**（弱きを助けるか）
- **忠義**（主・恩人への忠誠を貫くか）
- **野心**（自分が上に立ちたいか）
- **貪欲**（金銭・利得への執着）
- **血気**（怒りやすさ、暴力への閾値の低さ）
- **臆病**（危険回避傾向）
- **面子**（名誉・体面への執着）
- **情**（友情・恋愛・家族への重み）

価値観は行動の**重み付け**であり、行動そのものではない。魯智深型（義侠95・血気90・貪欲10）と林冲型（忠義90・面子70・血気30）は同じ状況で違う行動を選ぶ。

### 3.3 欲求（動的ドライブ）

生理系（食・眠・安全）と社会系（金銭・名誉・所属・愛情・復讐・報恩）の二層。
社会系欲求は**出来事によって生成・増幅される**。復讐欲は怨恨記憶から、報恩欲は恩義記憶から湧く。ここが感情システムとの接合点。

### 3.4 目標スタック（階層型）

- **人生目標**（価値観から導出。例：野心高→「一国一城の主になる」、義侠高→「不義を討つ」）
- **中期目標**（人生目標の分解。例：「仲間を集める」「拠点を得る」「仕官する」）
- **即時行動**（今日やること）

効用AIは「即時行動候補 × (欲求充足度 + 目標貢献度 + 価値観適合度 + 関係考慮)」でスコアリングする。ulo-aquariumで実証済みの欲求優先度設計の知見（優先度逆転による生存率崩壊の教訓）をそのまま持ち込む。

### 3.5 記憶

記憶エントリ = `{ 対象イベントID, 関与者, 感情価（正負）, 顕著度, 獲得tick }`

- **顕著度による容量管理**：武将一人あたり記憶上限（例：128件）。低顕著度から忘却。
- **重大記憶は不滅**：裏切り・命の恩・肉親の死などは忘却対象外。
- **想起トリガー**：関与者との再会、宴会、同地点への再訪で記憶が活性化し、行動候補（口論・報恩・回避）を生成する。

「誤射を宴会で蒸し返して口論になる」は、想起トリガー＋血気値＋酒（抑制低下バフ）の合成として自然に発生する。

## 4. 世界シミュレーション

### 4.1 地理モデル

マクロは**グラフ**、ミクロは**セル**の二層。

- **マクロ層**：地域ノード（都市・村・山塞・関所・渡し場）＋街道エッジ。移動・噂伝播・経済はグラフ上で動く。北宋華北〜江南を50〜80ノード程度に抽象化。梁山泊・開封・江州・大名府など水滸伝の要所を含む。
- **ミクロ層**：戦闘・遭遇時のみ展開されるセルグリッド（§7）。地形変化（焼失・落石）はマクロ層のノード属性へ書き戻され永続する。

### 4.2 人口の二層構造

- **名有り武将**：300〜600人をフル・シミュレート。108星＋官僚＋世代交代で生成される新規武将。
- **群衆**：ノード単位の集計値（人口・民心・治安・富）。名有り武将の行動（略奪・善政・戦争）が集計値を変え、集計値が名有り武将の環境（募兵可能数・盗賊発生率）を決める。

世代交代：死亡・引退による退場と、子・弟子・在野からの登場で人口を維持する。**百年回すには世代交代が必須**。

### 4.3 経済・官府

経済は物語の燃料としてのみ存在させる。詳細な市場シミュレーションは作らない。
- ノードごとの富と税。重税→民心低下→盗賊化→山塞形成、の一本道があれば水滸伝は駆動する。
- **官府（朝廷）は特殊な常設勢力**として存在し、腐敗度パラメータを持つ。腐敗が高いほど冤罪イベント・重税イベントが発生し、良い武将が野に落ちる。水滸伝の物語エンジンの核は「体制が人材を追放する」構造であり、これを世界の恒常圧力として実装する。

### 4.4 世界の攪拌装置

創発世界は放置すると平衡（一強停滞 or 全滅）に収束する（§16-5）。攪拌装置を世界側に常備する：
- 天災（洪水・飢饉）→ 流民 → 盗賊 → 新勢力の種
- 朝廷の粛清・佞臣の台頭 → 有能武将の追放
- 外圧（遼・金の侵攻）→ 大戦期の到来
- 代替わり → 勢力の内紛

## 5. 人間関係システム

### 5.1 有向・非対称・多軸エッジ

関係 = A→B の有向エッジ。**AがBをどう思うかとBがAをどう思うかは別物**。

軸：好感 / 信頼 / 敬意 / 恐怖 / 恩義台帳 / 怨恨台帳

### 5.2 恩義・怨恨は「台帳」である

数値ではなく**エントリの列**として保持する。
`怨恨エントリ = { 原因イベントID, 重さ, 未清算フラグ }`

- 数値なら「なぜ恨んでいるか」が消えるが、台帳なら物語システムがそのまま読める。
- 復讐・報恩の行動はエントリを「清算」し、清算自体がまたイベントになる。
- 台帳こそが因果保存原則の関係システム側の実装である。

### 5.3 関係ラベルの創発

義兄弟・親友・仇敵・師弟・恋人は**閾値＋儀式イベントで成立するラベル**。
例：好感・信頼が閾値超 ＋ 共闘経験 → 「結義」行動候補が発生 → 実行されると義兄弟ラベル成立（これ自体が高顕著度イベント）。
ラベルは行動の重みを恒久的に変える（義兄弟の危機→救出欲求が最優先化）。

### 5.4 噂システム（情報の非対称性）

武将は**真実ではなく、知っている情報**に基づいて行動する。
- 出来事は目撃者を持つ。目撃者から噂として街道グラフを伝播する。
- 伝播ごとに歪み確率（誇張・主語の誤り）。「Aが誤射した」が「Aが故意に射た」に化ける。
- 誤解に基づく怨恨・誤解の解消・讒言（意図的な偽噂）が全て同一機構で表現できる。

噂は物語生成の最強の火種だが、複雑性コストが高い。**Phase1は「歪みなし伝播」で開始し、歪みは Phase1.5 で導入**する段階計画とする（§16-2）。

## 6. 感情システム

### 6.1 二層構造：気分と感情記憶

- **気分（短期）**：怒・喜・恐・哀の一時値。行動選択の重みを数日間変える。酒・宴会・勝利・負傷で変動。
- **感情記憶（長期）**：§3.5の記憶＋§5.2の台帳。人生を規定する。

### 6.2 評価（Appraisal）モデル

出来事発生時、関与者・目撃者それぞれが評価を行う：

```
評価(武将, イベント) =
  自分への利害 × 当事者との関係 × 価値観フィルタ
  → 気分変動 + 記憶登録 + 台帳記帳 + 関係軸の増減
```

同じ「上司が部下を処刑した」イベントでも、忠義型は「已むを得ず」、義侠型は「非道」と評価し、後者にのみ怨恨が記帳される。**同一イベントが武将ごとに異なる意味を持つ**ことが、世界の多様性を生む。

### 6.3 感情は行動候補を生む

感情システムの出口は表情ではなく行動である。
怨恨台帳 → 復讐欲求 → 「襲撃」「讒言」「離反」「決闘」候補。
恩義台帳 → 報恩欲求 → 「馳せ参じる」「庇う」「身代わり」候補。
候補の採否は価値観次第（臆病な者は恨んでも動かない。それも人生である）。

## 7. 戦闘システム

### 7.1 設計思想：戦闘は「イベント生成器」

戦闘の出力はダメージではなく、以下の**世界向けイベント群**である：
- 死亡・負傷・捕縛（人間関係と勢力構成を変える）
- 誤射・誤爆・見殺し・救出（感情台帳へ記帳される）
- 地形変化（延焼・落石が世界地図へ永続書き戻し）
- 武勇伝（噂として伝播し、名声を変える）
- 撤退・寝返り（勢力図を変える）

ダメージと勝敗は副産物であり、**戦闘の真の成果物は「因果イベントの束」**である。

### 7.2 戦場モデル

- マクロ地域ノード上で戦闘が発生すると、セルグリッド（例：40×40）へ展開。
- ユニット = 名有り武将 ＋ 率いる兵（集計値）。
- 状態：位置・向き・士気・陣形・隠密。HPではなく**士気崩壊と武将の身体状態**が決着条件。
- 戦闘もイベント駆動：技・移動・士気変化すべてが同一イベントバスを流れる。

### 7.3 技 = 世界へ副作用を持つイベント発生器

| 技 | 直接効果 | 世界への副作用 |
|---|---|---|
| 突撃 | 敵を吹き飛ばす | 陣形崩壊、押し出された兵が第三者部隊へ衝突 |
| 斉射 | 矢の弾幕（各矢を個別解決） | 外れた矢が**任意の第三者（味方・非戦闘員含む）へ命中し得る** → 誤射イベント |
| 火計 | セル延焼（風向で拡大） | 焼失セルが地域ノードへ書き戻され、村の焼失・流民発生 |
| 妖術 | 複数セルへ恐慌 | 目撃者に「妖人」の噂、宗教的畏怖 |
| 落石 | 地形変化 | 街道封鎖が世界の移動グラフを恒久変更 |
| 伏兵 | 森へ潜伏 | 発覚時「卑怯」の評価が価値観次第で分かれる |
| 挑発 | 対象AIの目標を書換え | 挑発に乗った事実が記憶され、後日の面子問題に |

**「各矢を個別解決する」設計判断が重要**。弾幕を集計ダメージにすると誤射という物語の火種が死ぬ。矢は少数の抽象「矢束」として解決し、外れ矢ごとに落着セルを求め、そこに誰かがいれば命中判定する。

### 7.4 戦闘の終わり方

殲滅は稀。士気崩壊 → 潰走 → 追撃 or 見逃し。捕縛 → 処刑/登用/釈放は捕縛者の価値観と関係で決まり、いずれも重い感情イベントになる（釈放された者の恩義、処刑された者の縁者の怨恨）。

## 8. 勢力システム

### 8.1 勢力 = 武将の欲望の器

勢力に独立したAIは無い。方針 = 当主の目標 ＋ 重臣たちの進言（各自の欲望の反映）＋ 合議の摩擦。
当主が野心型なら拡張し、義侠型なら替天行道を掲げ、貪欲型なら略奪国家になる。

### 8.2 勢力の類型

| 類型 | 領地 | 例 |
|---|---|---|
| 朝廷（官府） | 全土名目支配 | 常設。腐敗度で世界を攪拌 |
| 地方軍閥 | 都市 | 大名府梁中書など |
| 山塞 | 山ノード（非都市） | 梁山泊・二龍山・桃花山 |
| 放浪勢力 | **無し** | 流賊、義兄弟団 |
| 結社 | 無し（潜伏） | 宗教結社、盗賊ギルド |

**領地は勢力の必須条件ではない**。これがSAN9PKとの最大の構造差であり、水滸伝化の要である。

### 8.3 ライフサイクル

結成 → 拡大 → （内紛 | 招安 | 討伐）→ 滅亡 → **残党（放浪勢力へ降格）** → 再興 or 消滅

- 結成条件：発起人＋同志3名以上＋結成動機イベント（冤罪・復讐・野心の成熟など）。**全武将が独立可能**。
- 滅亡は消滅ではない。生き残った武将は関係と記憶を保持したまま放浪へ戻る。「昔は放浪者だったのに今では国を興している」は、このループの2周目として自然に発生する。
- 内紛：怨恨台帳・野心・派閥（関係グラフのクラスタ）から発生。粛清・分裂・下剋上。

### 8.4 招安（体制への帰順）

水滸伝固有の名イベント。朝廷は強い山塞に対し討伐と招安の二択を持つ。招安受諾は勢力内の価値観分布で割れ（忠義派vs義侠派）、受諾しても腐敗した朝廷に使い潰される展開が構造的に発生し得る。悲劇の量産装置として意図的に設計へ含める。

## 9. 放浪システム

### 9.1 放浪は「状態」であり「物語の苗床」である

無所属武将の状態機械：

```
放浪（単独） → 遭遇 → 同行（党） → 逗留/仕官/入山/結義/決裂 → …
```

- 放浪者は街道グラフを目的地志向（仇を追う、恩人を訪ねる、仕官先を探す）＋漂泊で移動。
- **遭遇が物語の主生成源**：同一ノード・同一街道エッジに居合わせた武将間で遭遇判定。酒場・渡し場・護送路は遭遇率ボーナス地形。
- 遭遇の解決は両者の価値観・関係・状況の関数：意気投合／喧嘩／勝負→敬服／見て見ぬふり／強盗。

### 9.2 水滸伝的定型の創発可能性

野猪林（護送中の暗殺を友が阻止）を分解すると：
冤罪イベント（官府腐敗）→ 流罪護送（移動イベント）→ 暗殺依頼（怨恨＋金銭）→ 友人の尾行（恩義＋情報）→ 街道遭遇 → 救出。
**全て既設計の部品で構成できる**。定型をスクリプトせず、部品の合成で「似た構造の別の物語」が無数に生まれることを目指す。

### 9.3 党（パーティ）

同行者は党を形成。党は勢力未満の関係集合であり、入山・結義・仕官の単位になる。党の頭領は名声と関係で自然決定（ulo-aquariumの主従パーティ設計の教訓：移動デッドロック対策を初期から仕様化）。

## 10. 物語パッケージシステム（本作の核）

### 10.1 原理：物語 = 因果グラフの部分グラフ

イベントログは因果リンクで巨大な有向非巡回グラフを成す。
物語抽出とは、**このグラフから「読める塊」を切り出す編集行為**である。

### 10.2 編纂パイプライン（年tickで駆動）

```
1. 種イベント選定
   顕著度上位のイベント（死亡・建国・裏切り・大戦・結義…）を種にする
2. 因果трав走
   種から causes[] を遡り、影響先を下り、減衰閾値まで収集
3. 凝集フィルタ
   主要人物の重複率・地理的連続性で無関係な枝を刈る
4. 弧（アーク）検出
   起（状況設定）承（対立の発生）転（転換点）結（清算）へ
   イベント型を写像。転と結が欠ける塊は「未完の物語」として保留
5. パッケージ化
   { タイトル / 参加人物 / 発生地域 / 期間 / 出来事列 / 会話 / 結末 } を生成
6. 人物年表へ登録
   参加人物全員の年表に相互リンク
```

### 10.3 タイトル生成（二段構え）

- **決定論層（必須）**：テンプレート合成。`{地名}の{戦い|変|乱}` `{渾名}の{反乱|離反|最期}` `{人名}{救出|暗殺|結義}事件`。渾名は素質・逸話から生成（豹子頭、花和尚に相当する称号生成はulo-aquariumのカタカナ称号生成の知見を転用）。
- **LLM装飾層（任意）**：Anthropic APIで題・地の文・会話文を文芸的に清書。**シミュレーション本体から完全隔離**し、装飾の有無で歴史は1bitも変わらない。オフライン実行・キャッシュ前提。

### 10.4 会話の扱い

会話は編纂時に捏造せず、**イベント発生時にテンプレート生成してペイロードに保存**する。性格×関係×状況をキーにした台詞テンプレート（血気型の恫喝、義侠型の啖呵…）。編纂時は保存済み台詞を引用するだけにする。これにより同じ事件はどの物語から参照しても同じ台詞になる（歴史の一貫性）。

### 10.5 物語の品質スコア

抽出候補は全て採用しない。スコアリングで選別する：
- 緊張曲線（顕著度の時系列が山なりか）
- 因果の意外性（遠い原因が結末に効いているか）
- 人物の変化量（地位・関係・所属の始点終点差）
- 既出物語との重複率（同じ枝の使い回しを抑制）

**「因果的に正しいが物語的に無意味」な塊を落とす仕組みが本システムの成否を分ける**（§17-2）。

## 11. UI設計

### 11.1 UIの正体：歴史書リーダー

本作のUIは操作盤ではなく**書見台**である。中核画面：

1. **世界地図（PixiJS）**：勢力色、武将の点、進行中イベントの波紋表示。時間制御（停止/1x/10x/100x/年スキップ）。
2. **歴史書ビュー**：編纂済み物語の一覧と本文。物語内の人名・地名は全てリンク。
3. **人物帳**：武将の人生年表。各項目クリックで該当物語へジャンプ（林冲の年表要件そのもの）。関係グラフ（義兄弟・仇敵の可視化）。
4. **勢力志**：勢力ごとの興亡記。
5. **注目（フォロー）機能**：任意の武将・勢力・地域を登録すると、関連イベント発生時に通知。眺めるゲームの導線の要。
6. **年代記タイムライン**：百年を俯瞰し、任意時点へリプレイジャンプ。

### 11.2 演出方針

進行フェイズ中、地図上の武将は自律移動し、吹き出しで短い台詞・感情アイコンを出す（SAN9PK進行フェイズの快楽の再現）。ただし演出は全てイベント購読による後付けであり、演出を切ってもシミュレーションは同一に進む。

## 12. アーキテクチャ設計

### 12.1 レイヤ構成（ulo-aquarium準拠の厳格分離）

```
Domain          … 世界・武将・イベント・物語の純粋ロジック。依存ゼロ
Infrastructure  … 永続化、シード乱数、LLMアダプタ、イベントログストア
Application     … Tickオーケストレータ、史官（編纂）、シナリオローダ
Presentation    … PixiJSビューア / CLIランナー（両者は対等なクライアント）
```

- 配線は Composition Root（`main.ts` / `cli.ts`）に集約。
- Domainは `document` も `window` も知らない。Node（tsx）でそのまま100年回る。
- 公開APIは凍結、`any`禁止、マジックナンバー禁止、ファイル責務コメント必須、`Math.random()`禁止（シード乱数のみ）— 既存プロジェクトの規約を全面継承。

### 12.2 パッケージ分割

```
@world/core     … Domain + Application（シミュレーション本体）
@world/chronicle… 物語編纂（coreのイベントログのみに依存）
@world/cli      … ヘッドレスランナー
@world/viewer   … PixiJSビューア（Vite）
@world/llm      … LLM装飾アダプタ（chronicleの後段、任意起動）
```

viewerを消してもcore＋cliで製品（歴史生成器）として完結する。**Phase1の成果物はcore＋chronicle＋cliである。**

### 12.3 決定論の守り方

- 乱数は用途別ストリーム（世界生成/行動/戦闘/遭遇…）に分離し、機能追加が既存ストリームを乱さないようにする。
- イベント処理順は tick 内で完全固定（武将ID昇順など）。
- 浮動小数の順序依存を避け、可能な限り整数演算。

## 13. ドメイン設計

### 13.1 集約（Aggregate）

| 集約 | 責務 | 主な不変条件 |
|---|---|---|
| World | 時間・地理グラフ・天候 | tickの単調増加 |
| Agent（武将） | 価値観・欲求・目標・記憶・身体 | 記憶上限、死者は行動しない |
| RelationGraph | 全有向エッジ＋台帳 | エッジはID参照のみ（Agent実体を持たない） |
| Faction | 構成員・拠点・国庫・方針 | 当主は構成員、領地ゼロは合法 |
| Party | 同行者集合 | 頭領一意 |
| Battle | 戦場セル・部隊・進行 | 終了時に必ず世界へイベント束を返す |
| EventLog | 全イベント（append-only） | 因果リンクは過去方向のみ |
| StoryShelf | 編纂済み物語・年表 | イベントログへの読取専用参照 |

### 13.2 集約間の結合規則

- 集約間参照は**IDのみ**。直接のオブジェクト参照禁止。
- 状態変更は「コマンド → ドメインイベント発行 → 購読側が自集約を更新」。RelationGraphはBattleを知らないが、誤射イベントを購読して台帳を記帳する。
- この規則により、新システム（例：宗教）はイベント購読の追加だけで既存集約へ波及できる。

### 13.3 中核型（概念定義。コードではない）

- `WorldEvent`: `{ id, tick, type, actors[], witnesses[], location, causes[], payload, salience }`
- `LedgerEntry`: `{ 種別(恩義|怨恨), causeEventId, weight, settled }`
- `Story`: `{ id, title, protagonists[], region, span, beats[](=イベントID列＋役割), outcome, quality }`

## 14. イベントフロー

### 14.1 イベント分類

- **行動イベント**（武将が起こす）：移動・会話・攻撃・結義・離反…
- **結果イベント**（行動の帰結）：命中・誤射・死亡・延焼・捕縛…
- **社会イベント**（集約が起こす）：建国・滅亡・招安・粛清・天災…
- **情報イベント**（噂）：上記の伝聞。真偽・歪みを持つ

### 14.2 誤射連鎖のフルトレース（設計検証用の基準シナリオ）

```
[戦闘] Aが斉射 →(結果) 矢束の一部が逸れる
→(結果) 味方Cへ命中(負傷) causes:[斉射]
→(反応) Cの評価: 自傷×A味方×C血気高 → 怨恨記帳 causes:[命中]
→(数年後・宴会) 想起トリガー: CがAと同席+飲酒
→(行動) C口論 causes:[怨恨記帳, 宴会]
→(結果) 決裂 → Cの所属満足度低下
→(行動) C離反 causes:[決裂]
→(社会) Cが敵勢力へ仕官
→(数年後) 両勢力開戦 → CはA討伐を進言 causes:[怨恨]
→(編纂) 史官が種=「AC対決」から遡行し
   物語『逸れ矢の十年』を抽出
```

このシナリオが**設計変更のたびに通ることを確認する受け入れテストの雛形**とする（ヘッドレス検証スクリプト文化の継承）。

### 14.3 バスの規律

- 同期配送・tick内固定順。非同期は導入しない（決定論のため）。
- イベントは不変。訂正は打消イベントで表現。
- 無限連鎖対策：反応が生む行動候補は必ず**翌tick以降**で実行（同tick再帰禁止）。

## 15. データフロー

```
シード
 └→ 世界生成 → 初期状態スナップショット
      └→ [毎tick] 状態 ⇄ イベントバス → イベントログ(append-only)
            └→ [年tick] 史官: ログ→因果グラフ→物語→StoryShelf

セーブ/ロード: スナップショット＋以降のログ差分。シード＋ログで完全リプレイ可
```

- 史官は低顕著度イベントを編纂後に間引き可能（**ただし物語が参照するイベントは永久保持**）。

## 16. 想定される問題点

### 16-1. 計算量とメモリ
- 関係グラフ：500人なら最大25万エッジ。全ペア保持は不要。**接触があった相手のみエッジ生成**＋疎行列で管理。
- 記憶：上限＋顕著度剪定で頭打ちにする（設計済み）。
- イベントログ：100年で数百万件規模になり得る。低顕著度の集約化（「小競り合い37件」への畳み込み）とセグメント圧縮が必須。
- 日tick×500人：効用評価の軽量化（候補の事前絞り込み、時分割）。**Phase1最初の四半期に性能予算（100年/5分以内など）を数値で確定させるベンチ計画を組む。**

### 16-2. 噂システムの複雑性爆発
情報の非対称性は物語の宝庫だが、「各武将が世界の何を知っているか」の管理は状態量・デバッグ難度ともに重い。
→ 段階導入：Phase1=全員が真実を即知る（歪みなし）／Phase1.5=伝播遅延／Phase2=歪みと讒言。

### 16-3. 決定論とLLMの緊張
LLM出力は非決定的。→ 装飾層への完全隔離（設計済み）に加え、**装飾結果をイベントID keyed でキャッシュし、セーブデータに同梱**して再読時の文面揺れを防ぐ。

### 16-4. 創発バグのデバッグ不能性
「30年目に全武将が餓死する」型のバグは原因が20年前にある。
→ 因果リンクはバグ調査にも使える。イベントIDから因果祖先を全表示する**考古学ツール（CLI）**をPhase1初期に作る。ulo-aquariumの欲求優先度逆転事故の教訓：**世界健全性メトリクス（生存率・勢力数・放浪者比率・戦争頻度）を毎年ログし、閾値逸脱で警報**を出す。

### 16-5. 平衡収束（世界の熱的死）
放置シミュレーションは「一強停滞」か「全滅」に落ちやすい。§4.4の攪拌装置に加え、収束検知（勢力数・イベント多様性の低下）をメトリクス化し、攪拌強度をデータ調整できるようにする。ただし**個々の武将の意思決定へ神の手を入れることは禁止**（介入は環境側のみ）。世界の面白さをパラメータ調整で守り、武将の自律性で嘘をつかない、という線引き。

### 16-6. 物語の質の分散
抽出物語の大半は凡庸になる（§17-2）。品質スコアは必須だが、スコア関数自体のチューニングに人手評価の反復が要る。

### 16-7. 開発規模
Phase1だけで中規模プロジェクト。→ §18-3の完成定義とマイルストーン分割で管理。

## 17. 批判的レビュー：このゲームは本当に面白いのか

擁護せず、成立条件を疑う。

### 17-1. 「眺めるだけ」は本当に持つのか
Dwarf FortressのLegendsモード、RimWorldの物語生成、Crusader Kingsの人間ドラマ——先行例はどれも愛されているが、**プレイヤーの介入や利害があって初めて物語が「自分事」になる**という共通構造を持つ。純観察のLegendsモードを何時間も読むのは少数派である。
Phase1は「誰の物語でもない物語」を読ませることになる。これは文芸作品を読む行為に近く、**物語の質がそのまま製品の質になる**。シミュレーションがどれほど精緻でも、出力テキストが平板なら退屈なスクリーンセーバーである。
→ 結論：Phase1の主戦場はシミュレーションの深さではなく**編纂と読書UXの質**。§11の注目機能・年表リンク・タイトル生成は飾りではなく生命線。

### 17-2. 創発の連鎖は「因果的に正しいが、物語的に無意味」になりがち
誤射→怨恨→離反、は設計上つながる。しかし現実の出力の9割は「Aが移動した。Bと会った。何も起きなかった。」の列になる。創発系の宿命として、**ドラマは稀にしか起きず、起きても埋もれる**。
→ 品質スコア（§10.5）だけでは足りない可能性が高い。§18-1で対策を再設計する。

### 17-3. 水滸伝である必然性が弱い
現設計は汎用歴史生成エンジンであり、水滸伝は地名と称号のスキンに見える。水滸伝の魂は「腐敗した体制が好漢を野に追い、彼らが義で結ばれる」構造にある。§4.3の官府腐敗と§8.4の招安を入れたが、**冤罪→流罪→入山の圧力が世界の主旋律として十分強く鳴るか**はバランス次第であり、設計だけでは保証できない。
→ 世界健全性メトリクスに「体制起因の没落数／義による結合数」を含め、水滸伝らしさを計測可能にする。

### 17-4. 「勝たないゲーム」の市場は狭い
これは事実として受け入れる。ただし本プロジェクトの自己定義は「新ジャンルの歴史生成エンジン」であり、水滸伝は最初の世界に過ぎない。エンジンとして成立すれば三國志世界・オリジナル世界への転用が効く。**core＋chronicle＋cliをエンジン製品として独立させる§12.2の分割は、この賭けのヘッジでもある。**

### 17-5. Phase2（プレイヤー参加）が設計原則と矛盾する危険
プレイヤー武将も一エージェントとして扱えば原則は守れるが、プレイヤーは必ず「自分中心の面白さ」を要求する。テンポ（放浪の待ち時間）、死（プレイヤー武将の病死）、無力（歴史に影響できない）——観察ゲームの美点はプレイ体験としては苦痛になり得る。
→ Phase2の初期形態は「操作」ではなく**「憑依観察」（一武将の視点で世界を読む）＋限定的な意思表明（重大選択のみ介入）**に留めるのが原則と両立する最小形。本格操作は Phase2 の後半課題として切り離す。

## 18. 設計の改善案（自己レビュー3回）

### 自己レビュー第1回：物語の質問題（§17-2）への構造的対策

**指摘**：品質スコアは「選別」であり、母集団が凡庸なら選別しても凡庸。
**改善**：
1. **遭遇の劇的バイアス（環境側介入）**：遭遇判定の抽選重みに「劇的ポテンシャル」（両者間に未清算台帳がある／義兄弟の敵がいる／指名手配中）を加味する。仇同士が同じ酒場に居合わせる確率を素の偶然より上げる。武将の意思決定には触れない（§16-5の線引きを守る）ため、決定論と自律性を壊さずドラマ密度を上げられる。講談の「奇遇」はジャンルの約束事であり、水滸伝原典も偶然の再会で駆動している。
2. **未完の物語の追跡**：弧検出（§10.2-4）で「転」まで来て「結」が無い塊を史官が追跡リスト化し、関与者の遭遇バイアスへフィードバックする。伏線が回収されやすい世界になる。
3. **凡庸イベントの編纂時圧縮**：「三年の平穏」を一行に畳む要約規則を編纂側に持たせ、読書テンポを守る。

### 自己レビュー第2回：史官の単一視点問題

**指摘**：§10の史官は中立の全知視点であり、出力が「正史」一種類しかない。読み物としての奥行きに欠け、噂システム（歪み）の投資も回収しきれない。
**改善**：
1. **史官を世界内エージェント化**：史官は特定勢力に属する記録者であり、**知っている情報（噂含む）と立場のバイアスで歴史を書く**。梁山泊側の史書と官府の実録で同じ事件の記述が食い違う。
2. プレイヤー（読者）は複数の史書を突き合わせて真相（イベントログ＝神の記録）に迫れる。「歴史を読む楽しさ」が「歴史を検証する楽しさ」へ拡張される。
3. 実装上は同一編纂パイプラインに「情報フィルタ＋評価語彙テーブル」を差すだけで済み、増分コストが小さい。
→ ただしPhase1初期からは持たない。**Phase1は中立史官のみ、複数史観はPhase1.5**（噂の歪み導入と同期）に置く。

### 自己レビュー第3回：Phase1の完成定義が曖昧

**指摘**：「100年回しても面白い」は検証不能。ulo-aquariumの納品文化（型検査＋ヘッドレス検証）に照らすと、完成条件の数値化が欠けている。
**改善——Phase1 Definition of Done**：
1. **健全性**：任意シード10本で100年完走。全滅・一強停滞・人口崩壊なし（メトリクス閾値内）。
2. **基準シナリオ**：§14.2の誤射連鎖を含む代表因果連鎖5種（誤射→離反、冤罪→入山、恩義→馳せ参じ、内紛→分裂、滅亡→再興）が、100年中に統計的に有意な頻度で自然発生する。
3. **物語出力**：100年あたり編纂物語60〜120本、うち品質スコア上位20本を人手評価し、「読んで筋が追え、因果に驚きがある」評価が過半。
4. **性能**：ヘッドレス100年をローカルで所定時間内（ベンチ確定後に数値化）。
5. **考古学ツール**：任意イベントの因果祖先を辿れるCLIが動作。
これらをマイルストーンに割る：M1 世界骨格＋武将AI＋放浪遭遇 / M2 関係・感情・台帳＋基準シナリオ2種 / M3 勢力興亡＋戦闘 / M4 史官＋物語棚 / M5 バランス＋DoD検収。

### 自己レビュー第4回（追加）：戦闘の粒度過剰リスク

**指摘**：セル戦闘（§7.2）はPhase1では重装備すぎる恐れ。物語に必要なのは戦闘の**内部**ではなく戦闘が吐く**イベント束**である。
**改善**：戦闘を二段実装にする。Phase1前半は「抽象戦闘」——戦力・地形・武将特性から結果イベント束（死傷・誤射・武勇・延焼…）を確率生成する解決器。セル戦闘はM3以降に同一インターフェース（イベント束を返す）の別実装として差し替える。**戦闘インターフェースの凍結（入力：参加勢力と地形／出力：イベント束）**により、差し替えが世界へ波及しない。これはビューアの戦闘演出をPhase2へ送る判断も可能にする。
