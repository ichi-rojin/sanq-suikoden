## 0. 位置づけと原則

### 0.1 位置づけ

本書は構造設計書§5（検証設計）の三層——固有名詞監査・創発検証・第二世界試験——を、
実装発注可能な粒度まで具体化する。M0ゲート「固有名詞監査の機械化」の仕様本体であり、
以後の全マイルストーンの**検収基準を入口で確定させる**文書である（G14の呼び時：全Mの入口と出口）。

### 0.2 原則（本書の背骨）

1. **検出器は読む。決して書かない。** 全検出器・監査・メトリクスは observatory の読取専用クエリ基盤上に実装する（憲法②観測不干渉の子）。エンジン側に検出器のためのフック・フラグを追加することを禁止する。**イベントログだけから判定できない検出器があるなら、それは検出器の欠陥ではなくイベント設計の欠落**であり、G11のイベントスキーマ設計への要求として返す（§9）。
2. **測るものと作るものを混ぜない。** 構造カタログは実装禁止（台本禁止則）だが、その読取専用の影である検出器は合法である。検出器の検出率を上げるためのエンジン改変は、環境ツマミ（G16の権限内）以外すべて違反。
3. **数字には必ずラベルを貼る。** 本書の閾値・目標帯はすべて「仮」であり、確定するマイルストーンを併記する。仮の数字は約束ではない。

### 0.3 用語の定義（G9用語集への登録候補）

| 用語 | 定義 |
|---|---|
| **検出器（Detector）** | イベントログ上のパターン照合クエリ。検出インスタンス（§2.2）を出力する |
| **監査（Audit）** | 静的検査（ソース走査）または実行比較検査（ログ一致）。合否のみを出力する |
| **メトリクス（Metric）** | 年次等の定期集計値の時系列。警報閾値と対で運用する |
| **試験（Trial）** | 人手または複数実行を要する検収手順。手順書と合格基準を持つ |

## 2. 検出器の共通仕様

### 2.1 検出器定義スキーマ（概念）

```
Detector = {
  id,                  // 例: DET-3-1（構造検出器）, DET-S1（基準シナリオ）
  対象,                // 九構造 or 基準シナリオ or アドホック
  入力: イベント型の列と causes[] 連結条件,
  照合規則,            // §3〜§4 の個別定義
  出力: DetectionInstance[],
  集計: { 発生数/年, 関与者多様性, 構造間接続 }
}
DetectionInstance = { detectorId, 関与者ID列, 期間(tick範囲), 経路イベントID列 }
```

- **経路イベントID列が必須**である点が本仕様の要。検出は「あった/なかった」ではなく、
  因果グラフ上の実経路として出力する。これにより検出結果はそのまま「なぜ？」ボタン・
  物語編纂・デバッグ考古学の入力になる（開発ツールと製品機能の基盤統一）。
- 検出器は決定論的であること：同一ログ→同一の検出インスタンス集合。
- 実装位置：`@world/observatory` の因果クエリ・型別インデックス上。coreへの依存追加は禁止（軸F、CQRS契約）。

### 2.2 三つの標準集計（全検出器共通）

1. **発生数**：100年あたりの検出インスタンス数。
2. **関与者多様性**：検出インスタンス群の主要関与者の異なり数／総関与者数。**毎回同じ顔ぶれなら固定化（台本混入または勾配の枯渇）の兆候**（構造§5.2）。
3. **構造間接続率**：あるインスタンスの経路終端イベントが、別インスタンスの経路始端の因果祖先である比率（例：追放→結義の連鎖率）。**九構造は接続してこそ歴史になる**（構造§3）ため、個別発生数より重要な健全性指標である。

## 5. 監査3種の機械化要件（M0ゲートの本体）

### 5.1 固有名詞監査（静的・CI必須）

- **禁止リストの生成元＝世界パックの語彙テーブル**（人名・地名・勢力名・渾名・制度実体名・技名・大義名）。パックが唯一の真実源であり、手書きの禁止リストを作らない（リストの腐敗防止）。
- 走査対象：エンジン層（core/chronicle/observatory/cli）の全ソースとコメント。viewer は対象外（表示用語はG7裁量）だが、パック以外からの固有名詞 import は依存規則（G10）で別途遮断。
- 一般語との衝突（例：「関」「済」等の一字地名が普通の語に含まれる）：**完全一致トークン照合を既定**とし、衝突が残る語は語彙テーブル側に照合キー（表記＋文脈タグ）を持たせて解決する。エンジン側に例外リストを置かない（例外の置き場所も境界の外＝パック側）。
- 合格＝検出ゼロ。**M0で稼働開始し、以後すべての検証ゲートで常時実行**。

### 5.2 不干渉監査（実行比較・M2から）

- 同一シード・同一年数で、(a) 無操作実行、(b) observatory クエリを大量発行しながらの実行（Phase1はUIが無いため、CLI経由の因果クエリ・検索・集計の乱発で代替する）の2本を回し、**イベントログのハッシュ一致**で判定。
- 将来ビューア実装後は (b) をUI操作リプレイに拡張（観測§9-3-5）。

### 5.3 決定論監査（実行比較・M1から）

- 同一シード二重実行のイベントログ完全一致（ハッシュ）。乱数ストリーム分離（世界§12.3）の退行検知を兼ねる。
- M4以降：LLM装飾の有効/無効の2実行でイベントログが一致すること（軸B「LLMを止めても歴史は同一」のリリース条件を機械化）。
- **決定論に影響し得る修正（乱数消費順の変化）を含むPRは、本監査の通過ログ添付を必須とする**（P7・P8の完了定義と接続）。

## 6. メトリクスの算出定義と警報

### 6.1 健全性メトリクス（世界§16-4の実体化。年次集計、M1から）

| 指標 | 算出 | 警報（仮、M1で確定） |
|---|---|---|
| 生存率 | 名有り武将の年初生存数/前年 | 人口が初期値の50%未満または150%超 |
| 勢力数 | 領地あり＋領地なし勢力の数 | 2未満（一強停滞）または初期値3倍超（崩壊的分裂） |
| 放浪者比率 | 無所属武将/名有り総数 | 5%未満（流動性の死）または60%超 |
| 戦争頻度 | 戦闘イベント数/年 | 0が5年連続（停滞）または人口比閾値超（殲滅螺旋） |
| イベント多様性 | 年間イベント型のエントロピー | 移動平均の単調減少が10年継続（収束検知） |

警報はログ出力と異変フィード（観測§2.4）の共通供給源とする——**開発用警報と製品機能の基盤統一**。

### 6.2 らしさメトリクス（パック§12の6指標）

算出定義のみ本書が与える（目標帯はパック§12が正、調整はG16専権）：
体制起因の没落数＝DET-3-1発生数／義による結合数＝DET-3-2＋DET-3-7発生数／逼上梁山比率＝
山塞加入イベントのうちDET-3-1経路を因果祖先に持つ比率／名声相関＝官途・江湖名声の年次相関係数
（CR-3実装後）／招安後悲劇率＝DET-3-5の消耗・再離反枝の比率／酒店渡し場事件比率＝遭遇起点
イベントの発生ノード型集計。
**運用上の線引き（軸A・パック§14-2）**：メトリクスは健康診断であって目標関数ではない。帯逸脱への対応はG16の環境ツマミのみ。検出器・メトリクスを直接参照するエンジンロジックの追加は違反（観測不干渉の内側版）。

### 6.3 運命多様性試験（パック§13の機械化範囲）

10シード×100年でS層12名の人生終着を分類（官途全う/入山/独立/横死/隠棲/その他）。
**機械化するのは分類と分岐カウントまで**（終着分類はイベントログから決定論的に判定可能）。
「10/10一致＝台本混入疑い」の監査発報も機械化。原因調査は人手（考古学CLI）。

## 7. 人手評価の運用設計（負債D-2への応答）

- **物語採点表（1本5分様式）**：③項目3段階——筋が追えるか（迷子にならず読了できた/部分的/不能）、因果の驚き（遠い原因が効いていた/普通/自明）、人物の変化（始点と終点で違う人生になっている/薄い/無い）——＋自由記述1行。M4冒頭で様式を凍結してから編纂実装に入る（評価様式が後出しだと編纂側がテストに教え込まれる）。

- 合格基準は世界§18-3のDoD（上位20本の過半合格）を踏襲。**評価者はProducer1名を想定し、1回の検収セッションを20本×5分＝100分以内に設計する**（Producer時間は最希少資源）。

## 9. イベントスキーマへの要求仕様（G11ラウンドへの先行入力）

検出器群がイベントログに要求する最小属性を列挙する。これは**CRではない**（イベントスキーマは
未設計・未凍結であり、変更要求の対象がまだ存在しない）。G11の設計ラウンドへの入力仕様であり、
採否の裁定はG11ラウンド内で行う。

1. 全イベント共通：`type / actors[] / witnesses[] / location / causes[] / salience / tick`（世界§13.3を追認）。
2. 台帳参照：清算イベントは causes に**台帳エントリの原因イベントID**を含むこと（DET-3-6の成立条件）。
3. 判決イベント：讒言・後ろ盾・腐敗度の寄与を payload に持つこと（冤罪の事後判定可能性。DET-3-1）。
4. 結義・救出・帰順・継承主張・大義掲揚は独立イベント型であること（行動の副作用に埋めない。埋めると検出器が状態推測を強いられ、決定論的検出が壊れる）。
5. 勢力の結成・滅亡・分裂は構成員リスト差分を payload に持つこと（DET-S5の同一人物追跡）。

## 10. 想定される問題点

1. **偽陰性（創発の変種を見逃す）**：照合パターンは「十分条件」で書くため、パターン外の面白い連鎖を数えない。→ 対策：顕著度上位イベントの因果塊を月次でサンプリングし、どの検出器にも掛からない高顕著度連鎖を「未分類連鎖」としてレポート。**未分類連鎖はカタログ拡張（G4）の入力**であり、ノイズではなく発見として扱う。
2. **偽陽性（緩いパターンが何でも拾う）**：特にDET-3-2（共闘→結義）は日常の同行を拾いかねない。→ 経路の顕著度合計に下限を置く（仮値、M3で調整）。
3. **閾値が仮値だらけ**：本書の数値は全て仮。→ 各表に確定Mを明記済み。確定時に本書を改訂し版数を上げる（凍結後はCR経由）。
4. **ログ肥大時の検出器性能**：100年×数百万イベントの照合コスト。→ observatory の型別・因果隣接インデックス（観測§6.2）前提で設計済み。検出器の実行は年次バッチで足り、tick同期不要。G12へインデックス仕様の整合を依頼。

## 11. 批判章（G17）

1. **検出器仕様は「テストに教える」圧力を生む**。九構造の発生数が検収に載った瞬間、実装者は検出器を通すことを目標にし始める——それは一段回りくどい台本執筆である。→ 応答：防波堤は三つ。(a) 調整はG16環境ツマミのみ（機械監視付き、軸A）、(b) 関与者多様性・構造間接続率という「台本では稼げない」指標を主指標に置いた、(c) 未分類連鎖レポート（§10-1）が「検出器の外」を常時可視化する。それでも圧力は消えない。**検収会議では発生数より多様性を先に読む**運用規律を凍結条件に含めよ。
2. **監査の網羅性への過信**。固有名詞監査は語彙テーブルに載った名詞しか検出できない。ロジックに埋まる「実質的な固有名詞」（梁山泊だけに成立する条件式）は素通りする。→ 応答：正しい。機械監査は必要条件であり十分条件ではない。実質的台本の検出は凍結レビュー（人間＋G17の帽子）の責務として残る。機械化の範囲を偽らないこと。
3. **本書自体の運用コスト**。検出器9＋シナリオ5＋監査3＋メトリクス11＋試験3。一人開発でこの検収装置群を維持できるのか。→ 応答：Phase割当で同時稼働数を絞ってある（M2時点で検出器4・監査3・メトリクス5）。それでも重い。**M2終了時に検収コストを実測し、過重なら削減CRを起案する**ことを負債として記帳する。

## 13. 二重検問への回答文

**問1（プレイヤーを削除しても面白いか）**：本書はプレイヤー不在で回る世界の「面白さの発生」自体を
計測対象にする装置群である。検出器が測る構造の自然発生・関与者の多様性・構造間の連鎖は、
すべて観客ゼロの世界で起きる出来事の性質であり、むしろプレイヤーを削除した状態でしか正しく
測れない（不干渉監査はその削除状態との一致を要求する）。本書はYESを前提にするのではなく、
YESであることを検証可能にする——それが存在理由である。

**問2（再現か、生成か）**：本書のどの検出器も、特定の人物が特定の運命を辿ることを照合しない。
照合するのは構造の型と統計的性質であり、運命多様性試験に至っては**分岐しないことを異常として
発報する**。個々の歴史は自由に、世界の基調だけを見張る——「基調は守り、運命は放す」の計測側の
実装が本書である。生成の側に立つ。
