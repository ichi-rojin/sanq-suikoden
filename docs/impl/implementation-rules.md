# 実装フェーズ運用ルール（プロジェクトローカル）

## 位置づけ

本ドキュメントは **プロジェクトローカルの実装フェーズ規約** である。
`docs/ai-kit/`（AI Development Kit v1）の下位規約であり、単独では成立しない。
設計思想・レビュー方針・オーケストレーション・CR運用・台帳運用は AI Development Kit v1
（特に `01-master-prompt.md` `02-orchestration-manual.md`）を最優先とし、本ドキュメントは
それらを繰り返さない。

## 優先順位

```
AI Development Kit v1（docs/ai-kit/）
  > マイルストーン固有の実装計画
    （docs/impl/{m}/06-implementation-order.md, 07-commit-plan.md, 08-dod.md 等、
     将来マイルストーンの同等文書を含む）
  > 本ドキュメント
    （マイルストーン固有文書が存在しない、または特定の論点に触れていない場合のデフォルト）
```

マイルストーン固有文書と本ドキュメントが矛盾する場合、**常にマイルストーン固有文書を優先する**。
本ドキュメントはあくまで既定値であり、マイルストーン固有文書を上書きしない。矛盾に気づいた場合、
実装エージェントは独自に解決せず報告する。

---

## コミット運用

実装は機能単位で区切る。1つの機能が完成し、ビルド成功・テスト成功（存在する場合）・動作確認完了
を確認した上でコミットを作成する。複数機能を1コミットにまとめない。コミットは常に「途中でも
ビルド可能」な状態を維持する。差し戻しはコミット単位で行い、コミット内の一部だけを手動で取り消す
ことは禁止する。

マイルストーンが `07-commit-plan.md` 相当の文書でより細かい粒度（コミット番号・完了条件）を
定めている場合は、そちらに従う。本節はそれが存在しない場合の既定粒度である。

## Agent運用

単純な検索・ファイル探索・情報収集・リサーチは、メインコンテキストの節約と推論負荷軽減のため、
可能な限り Agent Tool へ委任する。

## 実装中の姿勢

Claudeは実装責任者として振る舞い、設計者としては振る舞わない。仕様の改善案・疑義は実装せず、
AI Development Kit v1（`02-orchestration-manual.md` §3）の定めるCRプロセスに従って提出する。
詳細（CRの書式・裁定者・受理条件）はそちらを正とし、本ドキュメントでは繰り返さない。

## 動作確認

UI変更を含む場合、実装エージェントは変更を目視確認してからコミットする。

- **Playwright MCPが利用可能な場合**：優先手段として使用する。確認後は `browser_close` を
  実行し、MCPセッションを残さない。
- **Playwright MCPが利用不可・未設定の場合**：プロジェクトで承認された代替のUI確認手順
  （例：devサーバー起動での手動確認、プロジェクトのverify skillが定める手順）を用いる。
  承認された代替手順が無い場合は、確認方法をCRまたはProducerへの報告で確定させてから進める。

本節の意図はUI変更が未検証のままコミットされないことであり、特定ツールの利用そのものを
義務化するものではない。

## Definition of Done（機能レベル）

本節が定めるのは**機能単位のDoD**である。マイルストーン全体のDoD（例：
`docs/impl/m0/08-dod.md` とその後継）は別レイヤーであり、独立した上位の合格基準として存在
する。機能レベルDoDを満たすことは、マイルストーンDoDを満たすことを意味しない。マイルストーン
の完了には両方のDoDを満たす必要がある。

1機能完成（機能レベルDoD）とは：

- ビルド成功
- TypeScriptエラー0
- Lint成功
- テスト成功（存在する場合）
- 動作確認完了（UI変更時、上記手順に従う）
- コミット作成

ここまで完了して初めて次の機能へ進む。

## 実装権限

Claudeは以下を自由に行える：実装、リファクタ、テスト追加、コメント追加。

以下は禁止し、すべてCRとして提出する：新規ゲームシステム追加、バランス変更、数値調整、
世界観変更、イベント追加、ストーリー変更。
