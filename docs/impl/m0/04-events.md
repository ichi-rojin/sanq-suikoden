# イベント一覧 ―（成果物④）

## 0. M0の正直な回答

**M0の実行時に流れるイベントは存在しない。** M0はスキーマと監査の実装であり、シミュレーションは起動しない。
ドメインイベントの具体カタログ（行動・結果・社会・情報の4分類——世界設計書§14.1）は**G11イベントスキーマ設計ラウンドの成果物**であり、M1実装の前提として別途設計される。ここで具体イベント型を発明することは幽霊仕様（接続なき設計）になるため行わない。

M0が実装するのは以下の**型定義のみ**である。

## 1. Domain Event（M0: 封筒型のみ）

| 名前 | 種別 | 定義 | M0での扱い |
|---|---|---|---|
| `WorldEvent` | 封筒型（全イベントの共通形状） | `{id, tick, type, actors[], witnesses[], location, causes[], payload, salience}`（世界設計書§13.3の転写） | 型定義＋形状テストのみ。発火なし |
| `EventBus` | 配送インターフェース | `publish` / `subscribe`。配送規律（同期・tick内固定順・翌tick持ち越し・打消——世界§14.3）はM1実装 | インターフェース定義のみ |

**G11ラウンドへの申し送り**: 検出器仕様書§9の先行入力5項（台帳参照をcausesに含める・判決payload・結義/救出/帰順/継承主張/大義掲揚の独立イベント型化・勢力構成員差分payload）を、イベントカタログ設計の入力とすること。

## 2. Application Event

**なし。** M0のユースケース（LoadPackUseCase）は同期の関数呼び出しで完結し、イベントを介さない。CLIは戻り値（ValidationReport）を整形するだけである。バスを通す必然がない場所にイベントを増やさない（イベントは世界の変化の記録であって、関数呼び出しの代替ではない）。

## 3. UI Event

**なし。** M0のUIはCLI2コマンド（validate-pack / audit-lexicon）のみで、対話的UIは存在しない。

## 4. M0の「イベント的」出力（イベントではないが記録されるもの）

| 出力 | 発生条件 | 受信者 | 副作用 |
|---|---|---|---|
| `ValidationReport`（JSON） | validate-pack 実行時 | CLI標準出力・verify.sh | 終了コード（issues空=0、非空=1）。CI失敗 |
| `AuditReport`（JSON） | audit-lexicon 実行時 | 同上 | 同上 |

これらは観測・検査の報告であり、世界の出来事ではないため WorldEvent を使わない（用語の混同を用語集で禁止）。
