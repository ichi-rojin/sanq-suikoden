# プロンプト運用手順書（Runbook）v1 ― 各モデルへ流すプロンプトの順番

現在地: M0完了。M1はG11イベントスキーマ設計ラウンド待ち。
プロンプト原本: P1〜P12＝docs/ai-kit/04-prompt-library.md／Sprint実装プロンプト＝docs/impl/{m}/09。
モデル責務: docs/ai-kit/02-orchestration-manual.md §1（Fable=主任設計者、Opus=レビュー/主任代行、Sonnet=実装・データ職人、Claude Code=実装環境）。

## 前提（毎回共通）

- **Claude Codeのセッション**（Fable/Sonnet/Opusいずれを載せる場合も）: リポジトリのCLAUDE.mdが入口を自動供給するため、添付作業は不要。プロンプトを貼るだけ。
- **claude.aiチャットのセッション**（Opusレビュー等）: 01-master-prompt.md＋台帳4種＋対象文書を**手動添付**してからプロンプトを貼る。
- どのセッションも開始テンプレ（02§7）で帽子を宣言し、終了テンプレ（02§8）で締める。途中終了したら次セッション冒頭はP12（復旧）。
- どのモデルも、仕様の欠陥を見つけたら P4（CR起案）で停止。勝手な改善は禁止。

## 分岐時に使うプロンプト（常備・順不同）

- **バグ**: Claude CodeへP7（再現→考古学→三分類診断→最小修正→回帰防止）
- **リファクタリング**: Claude CodeへP8（ログ一致証明必須）
- **CR起案**: どのモデルでもP4 → 裁定はFable（P11が必要な規模なら所見統合会議）
- **セッション事故**: 次セッション冒頭にP12（復旧）

## 順番の背骨（迷ったらこれ）

```
設計が足りない → P1(Fable) → P2(Opus) → [P3] → P10凍結(Fable) → Producer承認
実装へ         → 実装準備(Fable) → Sprint実装(Sonnet/Claude Code) → 実装レビュー(Opus/CC)
検収           → 検証ゲート(CC=G14) → 失敗はP7/P4/P5へ分岐 → スプリントレビュー(Producer+Fable)
```

発注順序の原理は依存順（語彙→イベント文法→エージェント→社会→物語→観測。06-tacit-knowledge §5）。
