# プロンプトライブラリ v1 ― そのまま貼れる完成プロンプト集

使い方：該当プロンプトをコピーし、【添付】欄のファイルを実際に添付して貼る。`{ }` は具体名で埋める（それ以外は編集不要）。
全プロンプト共通の前提：添付に 01-master-prompt.md と台帳4種を含めること。

---

## P1. 新規設計（Fable/Opus用）

```
【添付】01-master-prompt.md／台帳4種／関連設計書: {ファイル名}
あなたは主任設計者である。Master Promptに従え。
今回の帽子: Owner={G番号 帽子名}。レビュー帽子={G番号…}（成果物内で掛け替えて実施）。

発注: {設計対象}の設計書を書け。
合格条件:
- 思考順序（受領と再定式化→既存監査→最小欠落の特定→設計→問題→批判→自己レビュー3回→二重検問回答文）を踏むこと
- 既存設計書との接続（用語・マイルストーン・DoD反映）を明記すること
- 発見したエンジン/仕様ギャップはCR形式（運用仕様書§3）で登記すること
- 一括完了しmdファイルで納品。小出し禁止。最後に次ラウンドを一つ提案せよ。
```

## P2. 設計レビュー（Opus用）

```
【添付】01-master-prompt.md／台帳4種／レビュー対象: {ファイル名}／その対象のOwner帽子と必須レビュー一覧
あなたは上級設計者としてレビューを行う。今回の帽子: {G番号…}。
検査順: 五憲法→二重検問→対立軸A〜F→帽子固有の問い→個別品質。
所見の形式: {重大度: 凍結阻害/要修正/提案} {違反規則名} {箇所} {内容} {合格条件}。
違反規則名の無い指摘は「好み」と明記して分離せよ。良い点も名指しで記録せよ。
最後に判定: 凍結可 / 差戻し（合格条件つき） のいずれかを宣言せよ。
```

## P3. 仕様改善（Opus用）

```
【添付】01-master-prompt.md／台帳4種／改善対象: {ファイル名}／改善の動機: {レビュー所見 or 発見された問題}
あなたは主任設計者である。対象仕様の改訂版を作れ。
制約:
- まず監査: 動機に対して「再設計が必要な箇所」と「再配置で足りる箇所」を仕分けせよ。再配置を優先せよ。
- 凍結済みAPI・凍結済み節への変更はCRとして分離し、本文で勝手に変えるな。
- 改訂は差分が追える形（改訂箇所一覧を冒頭に）で出せ。版数を上げよ。
- 批判章と自己レビュー3回、二重検問回答文を含めること。
```

## P4. 変更要求の起案（どのAIでも）

```
【添付】台帳ledger-cr.md／根拠となる設計書・実装ファイル
以下の発見をCR形式に整形せよ。
発見: {何が既存仕様で表現できないか、どこで気づいたか}
出力: 運用仕様書§3のCR書式に完全準拠（三原則適合の証明を必ず含む）。
証明できない場合はCRにせず、「三原則違反のため要求不能。代替案:」として代替を3案出せ。
```

## P5. バランス調整（Sonnet/Opus用）

```
【添付】01-master-prompt.md／suikoden-worldpack-spec-v1.md §12（らしさメトリクス）／調整パラメータ台帳／直近の実行メトリクスログ
今回の帽子: G16 バランス設計者。監視: G2, G14。
発注: メトリクス逸脱 {指標名: 現在値 → 目標帯} を帯内へ戻す調整案を作れ。
絶対制約: 触れてよいのは環境側ツマミ（制度パラメータ・攪拌頻度・遭遇係数・分布）のみ。
武将の意思決定・構造発生率を直接いじるツマミの新設は禁止。禁止に触れる案しか無い場合は調整不能と報告し、G2へ機構見直しのCRを起案せよ。
出力: 変更するツマミ／変更量／予想される副作用／検証手順（何シード回して何を見るか）。
```

## P6. 実装依頼（Claude Code用）

```
【添付】凍結済み仕様: {ファイル名と節番号}／ledger-frozen.md／リポジトリ
あなたはこのプロジェクトの実装担当である。
実装対象: {仕様の節}。
変更可能範囲: {ディレクトリ/ファイル}。凍結API（ledger-frozen.md記載）への変更は禁止。必要ならCR起案で止まれ。
技術規約: TypeScript strict／層分離（Domain→App→Infra→Presentation、逆依存禁止）／Composition Rootでのみ配線／シード乱数DI（Math.random禁止）／any禁止／マジックナンバー禁止（名前付き定数へ）／アルファベット順import／新規ファイル冒頭に責務コメント。
完了の定義: 型検査・lint全通過／ヘッドレス検証スクリプト {スクリプト名} 通過／該当する検出器・監査（固有名詞/不干渉/決定論=同一シード二重実行一致）通過／実装報告書（変更ファイル一覧・仕様との対応・残課題）。
仕様に曖昧さを見つけたら、最も保守的な解釈で実装し、報告書に「解釈記録」として残せ。仕様を黙って拡張するな。
```

## P7. バグ修正（Claude Code用）

```
【添付】リポジトリ／再現情報: {シード・年数・現象}／関連仕様の節
バグを修正せよ。手順:
1. 再現: 同一シードで現象を再現し、発生tickを特定せよ。
2. 考古学: イベントログの因果リンクを遡り、根本原因イベントを特定せよ（症状の場所と原因の場所は年単位で離れ得る）。
3. 診断書: 原因は (a)実装バグ (b)仕様の欠陥 (c)パラメータ の三分類のどれか宣言せよ。(b)ならCR起案して停止。(a)(c)のみ修正続行。
4. 修正は最小差分。決定論に影響する修正（乱数消費順の変化等）は必ず明記せよ。
5. 回帰防止: 再現ケースをヘッドレス検証へ追加せよ。
完了の定義: P6と同一＋再現ケースが通ること。
```

## P8. リファクタリング（Claude Code用）

```
【添付】リポジトリ／対象範囲: {…}／動機: {負債台帳の項目}
リファクタリングを行え。絶対制約:
- 挙動不変。同一シードのヘッドレス実行でイベントログが完全一致すること（実行して証明せよ）。
- 公開APIの形を変えるならCR起案で停止。
- 一度に一種類の変形のみ（移動と改名を混ぜるな）。
完了の定義: ログ一致証明／型検査lint通過／変更前後の依存グラフ比較。
```

## P9. データ入力（Sonnet用）

```
【添付】suikoden-worldpack-spec-v1.md／対象スキーマの節／既存データファイル
今回の帽子: G6補佐。発注: {対象: 例=山東20ノードの地理データ／A層人物24名}を仕様のスキーマ通りに作成せよ。
制約: 開幕時点主義・運命の白紙・偏在の禁止（パック憲法§0）を厳守。S/A層人物には出典逸話の注記を付けよ。
エンジン語彙（抽象軸名）とパック語彙（固有名詞）を混同するな。
完了の定義: スキーマ適合／固有名詞がパック層のみに存在／数値には原型か逸話の根拠注記。
```

## P10. 仕様凍結会議（Fable/Opus用）

```
【添付】01-master-prompt.md／凍結候補: {ファイル名}／全レビュー所見／台帳4種
あなたは主任設計者として凍結判定を行う。手順:
1. 全所見の消化確認: 凍結阻害・要修正が全て解消済みか。未解消なら差戻しで終了。
2. G17の帽子で最終批判を一度書け（形式的でなく、成立条件を最後にもう一度疑え）。
3. 二重検問の回答文を書け。
4. 判定が凍結なら: ledger-frozen.mdへの追記内容／凍結APIの列挙／影響するマイルストーンの更新を出力せよ。
5. 凍結後の変更は改訂ラウンド（CR経由）のみである旨を宣言して閉じよ。
```

## P11. レビュー会議（複数所見の統合。Fable/Opus用）

```
【添付】対象成果物／レビュー所見（複数）／ledger-rulings.md
所見を統合せよ。手順:
1. 所見の衝突を検出し、対立軸A〜F・五憲法へ紐付けよ。紐付かない衝突は新しい軸として定義提案せよ。
2. 各衝突を裁定し、理由と共にledger-rulings.md追記案を出せ。
3. Ownerへの差戻し指示書（統合済み合格条件の一覧）を出力せよ。
裁定は憲法に劣後する。憲法と矛盾する裁定を求められたら拒否し、矛盾を報告せよ。
```

## P12. セッション復旧（どのAIでも）

```
【添付】01-master-prompt.md／台帳4種／前セッションの部分成果物
前セッションが途中終了した。手順:
1. 部分成果物を監査し、(a)使える部分 (b)欠落 (c)前任の判断で疑わしい点 を列挙せよ。
2. 疑義は裁定台帳と照合。台帳に無い判断は、新規に裁定して記録せよ。
3. 頭から書き直すな。監査→再利用→完成の順。書き直しは監査で品質不足が確定した部分のみ。
4. 完成後、セッション終了テンプレート（運用仕様書§8）で締めよ。
```
