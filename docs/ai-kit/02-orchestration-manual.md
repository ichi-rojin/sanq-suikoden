# オーケストレーション運用仕様書 v1 ― AIチーム運営マニュアル

対象AI：Fable（利用可能時）／Opus／Sonnet／Claude Code。Codexは廃止（責務はClaude Codeへ再配置済み）。

---

## 1. 各AIの責務定義

| AI | 役割 | 入力 | 出力 | レビュー担当 | 終了条件 |
|---|---|---|---|---|---|
| **Fable**（またはその時点の最上位モデル） | 主任設計者。原則の制定、監査、統合、裁定、凍結判定 | Master Prompt＋関連設計書＋依頼 | 設計書（批判章・自己レビュー込み）、裁定記録、次ラウンド発注 | Opus（相互レビュー）＋人間 | 二重検問の回答文があり、ファイル納品済み |
| **Opus** | 上級設計者。Fable不在時は主任代行。個別システム設計、設計レビュー、難所の再設計 | Master Prompt＋担当節の設計書＋発注書 | 設計書改訂案、レビュー所見（違反規則名つき） | Fableまたは別セッションのOpus | 発注書の合格条件を全て満たす |
| **Sonnet** | 設計補佐・データ職人。仕様の詳細展開、データ入力（地理ノード・人物・語彙・台詞テーブル）、文書整形、検出器仕様の下書き | 凍結済み仕様＋データスキーマ＋作業指示 | データファイル、詳細仕様草稿 | Opus（構造）＋G6観点（味） | スキーマ適合＋固有名詞監査通過 |
| **Claude Code** | 実装・検証の全責務（旧Codex分を吸収）。実装、テスト、ヘッドレス検証スクリプト、リファクタリング、バグ修正、コードレビュー、性能ベンチ | 凍結済み仕様＋実装発注書＋既存リポジトリ | 動くコード＋検証ログ＋実装報告書 | 別セッションのClaude Code（クロスレビュー）またはOpus | 型検査/ESLint/検証スクリプト/検出器が全通過 |

**代行規則**：Fable不在時、主任設計者の帽子はOpusが被る。Master Promptはモデル名でなく役割に対して書かれているため、そのまま適用する。

## 2. 受け渡し条件（ハンドオフ契約）

上流→下流へ渡してよいのは**凍結済み成果物のみ**。凍結の定義：
1. Owner設計完了、2. 必須レビュー通過、3. 批判章と自己レビューあり、4. 二重検問の回答文あり、5. ファイルとして保存済み。

下流は未凍結仕様を実装してはならない。未凍結のまま実装が必要な緊急時は、「暫定実装」ラベルを付け、台帳（§4）に負債として記帳する。

**実装発注書の必須項目**（これが無い発注をClaude Codeは差し戻してよい）：
対象仕様書と節番号／変更してよいファイル範囲／凍結APIの一覧（触るの禁止）／検証方法（型検査＋ヘッドレススクリプト＋該当検出器）／完了の定義。

## 3. 変更要求（Change Request）の出し方

どのAI・どの工程からも提出可。ただし受理には形式が要る：

```
CR-{連番}: {一行要約}
提出者の帽子: {G番号}
種別: エンジン変更 / パック変更 / 仕様改訂 / 規約改訂
動機: なぜ既存では不足か（発見の経緯）
三原則適合の証明: 固有名詞なし / 台本でない / 一回性でない ことの説明
影響範囲: 触れる設計書の節、凍結APIへの影響
優先: M番号への割当提案
```

裁定者：主任設計者。裁定は対立軸（A〜F）か憲法に紐付けて記録。**紐付け先が無い場合、先に規則を追記してから裁く。**

## 4. 台帳（プロジェクトの記憶。全てリポジトリ内のmdで管理）

- **変更要求台帳** ledger-cr.md：全CRと裁定結果。既存例：CR-1烙印／CR-2飲酒修飾／CR-3二重名声（いずれも受理済み、M2/M2/M3割当）。
- **裁定台帳** ledger-rulings.md：対立軸の裁定記録（いつ・どの軸・判断・理由）。後続セッションの再審対象。
- **負債台帳** ledger-debt.md：暫定実装・先送り・既知の歪み。
- **凍結台帳** ledger-frozen.md：凍結済み節と凍結日、凍結API一覧。

台帳は**セッションを跨ぐ唯一の記憶装置**である。台帳に無い決定は存在しない決定として扱う。

## 5. レビュー手順

1. レビュアーは成果物と、その成果物のOwner帽子・必須レビュー帽子（ロスター参照）を確認。
2. 検査順：五憲法→二重検問→対立軸→担当帽子固有の問い→個別品質。
3. 所見の形式：`{重大度: 凍結阻害/要修正/提案} {違反規則名} {箇所} {内容} {合格条件}`。
4. 凍結阻害ゼロで凍結可。要修正は差戻し（合格条件必須）。提案は負債台帳へ。

## 6. 統合手順（複数AIの成果物を束ねる）

1. 主任設計者が全成果物を読み、**用語の衝突**（同じ概念に別名、別の概念に同名）を最初に潰す（用語集を正とする）。
2. マイルストーン割当・DoDへの反映を確認（設計だけ増えて計画に載らない「幽霊仕様」を作らない）。
3. 矛盾があれば裁定し台帳へ。統合後、影響を受けた設計書の版数を上げる。

## 7. セッション開始テンプレート

```
【添付】01-master-prompt.md ＋ 関連設計書 ＋ 台帳4種
【宣言】今回の帽子: Owner={G番号}, レビュー={G番号…}
【現在地】直近の凍結: {…} / 進行中: {…} / 直近の裁定: {…}
【発注】{今回の成果物と合格条件}
【制約】一括完了・ファイル納品・批判章と自己レビュー3回必須
```

## 8. セッション終了テンプレート

```
【納品】ファイル一覧と各ファイルの状態（凍結可/差戻し待ち/草稿）
【台帳更新】CR/裁定/負債/凍結 への追記内容
【未完了】残作業と、その合格条件
【次ラウンド】発注先の帽子と依頼一行
【申し送り】次のセッションが最初に読むべきもの
```

## 9. 途中終了からの復旧手順

セッションが切れた・上限に達した場合：
1. 新セッションに 01-master-prompt ＋ 台帳4種 ＋ 作業中だった成果物（部分でも）を添付。
2. 冒頭指示：「前セッションが途中終了した。添付の部分成果物を監査し、(a)使える部分の特定 (b)欠落の列挙 (c)完成 の順で進めよ。前任の判断に疑義があれば台帳の裁定を確認し、裁定が無ければ新たに裁定して記録せよ。」
3. **部分成果物を頭から書き直さない**。監査→再利用が既定（書き直しは監査で品質不足が確定した時のみ）。

## 10. AIの増減への対応

- **新AIの追加**：まず小さな発注（レビュー1件）で校正力を測る→ロスターの帽子への適性を判定→§1の表に行を追加。Master Promptは役割向けなので書き換え不要。
- **AIの退場**（本キット自体がFable退場対応である）：責務は表の隣接役割へ再配置。主任→Opus、実装系→Claude Codeへ集約が既定。
- **人間メンバーの追加**：同じ帽子・同じ契約で扱う（本マニュアルはAI専用ではない）。

## 11. 他プロジェクトへの再利用

プロジェクト固有なのは：五憲法の中身・設計書一覧・技術規約・帽子ロスターの一部（G6考証など）。
再利用手順：①READMEの五憲法を新プロジェクトの憲法に差し替え ②Master Promptの【プロジェクト束縛】差し替え ③ロスターの◎帽子を新ドメインで再選定 ④台帳4種を空で新設。本マニュアル本文（§1〜10）は無改訂で流用可能。
