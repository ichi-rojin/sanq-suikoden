# 開発フローチャート v1 ― 完成までの運用フロー

## 0. 全体構造：スプリント＝ラウンドの束

プロジェクトはマイルストーン（M0〜M5, P1.5）を目標に、スプリント単位で進む。
1スプリント = 設計ラウンド（0〜n個）＋実装ラウンド（0〜n個）＋検証ゲート。

```
Producer(NERD)
   │ ①発注（対象と合格条件）
   ▼
[設計ラウンド] Fable/Opus（Owner帽子で設計）……P1
   │ 納品（批判章・自己レビュー込み）
   ▼
[レビュー] Opus（別セッション推奨）……P2
   │
   ├─ 凍結阻害あり ──▶ 差戻し（合格条件つき）──▶ 仕様改善 P3 ──▶ 再レビュー
   ├─ 所見が衝突 ──▶ レビュー会議 P11（裁定→台帳）──▶ 差戻し or 凍結判定へ
   ▼
[凍結判定] Fable/Opus ……P10
   │ 凍結（ledger-frozen更新）
   ▼
[実装発注書作成] 主任設計者
   │ （対象節・変更可能範囲・凍結API・検証方法・完了定義）
   ▼
[実装] Claude Code ……P6 ／ [データ入力] Sonnet ……P9
   │
   ├─ 仕様の曖昧さ発見 ──▶ 保守的解釈＋解釈記録（続行）
   ├─ 仕様の欠陥発見 ──▶ CR起案 P4 ──▶ 裁定 ──▶ 受理なら設計ラウンドへ戻る／却下なら続行
   ▼
[実装レビュー] 別セッションのClaude Code or Opus
   │ 規約・依存・決定論の検査
   ▼
[検証ゲート] G14の帽子（Claude Codeが実行）
   │ 型検査／lint／ヘッドレス検証／検出器／監査（固有名詞・不干渉・同一シード一致）／性能ベンチ
   │
   ├─ 失敗（実装起因）──▶ バグ修正 P7 ──▶ 再検証
   ├─ 失敗（仕様起因の診断）──▶ CR起案 ──▶ 設計ラウンドへ
   ├─ 失敗（数値起因）──▶ バランス調整 P5 ──▶ 再検証
   ▼
[スプリントレビュー] Producer + 主任設計者
   │ 台帳4種の更新確認／メトリクス確認／次スプリント発注
   ▼
次スプリントへ（→①）
```

## 1. 分岐規則の詳細

- **差戻しは2回まで**。3回目の差戻しが発生したら、発注自体に欠陥がある（合格条件が曖昧／対象が大きすぎる）と見なし、Producerと主任設計者が発注を分割・再定義する。
- **CRの流れ**：起案（P4）→ 主任設計者が三原則適合を審査 → 受理なら優先度（M番号）を付けて台帳へ → 該当スプリントで設計ラウンド化。却下時も台帳に残す（同じ要求の再発明を防ぐ）。
- **緊急時の暫定実装**：凍結を待てない場合のみ、「暫定」ラベル＋負債台帳記帳で実装可。次スプリントの冒頭で必ず清算対象として審議する。
- **凍結の遡及変更**：禁止。必要ならCR→改訂ラウンド→版数更新→再凍結。

| M | 設計:実装:検証の比重 | 検証ゲートの追加項目 |
|---|---|---|

| M1 | 4:4:2 | 性能予算の数値確定、observatory因果クエリ稼働 |
| M2 | 3:4:3 | 基準シナリオ2種（誤射連鎖・追放）検出器で確認 |

## 3. セッション運用の一日単位フロー

```
開始 ─ セッション開始テンプレート（帽子宣言・現在地・発注）
   ─ 作業（該当プロンプトP1〜P12）
   ─ G17の帽子を最低一度被る（毎セッション義務）
   ─ セッション終了テンプレート（納品・台帳更新・申し送り）
終了 ─ 上限や途中終了の場合 → 次セッションでP12（復旧）
```

## 4. Producerの関与点（NERDが判断すべき箇所）

Producerの時間は最希少資源として扱い、関与点を4つに限定する：
1. スプリント発注（何を作るか・合格条件の承認）
2. 凍結の最終承認（主任設計者の凍結判定に対する拒否権）
3. 人手評価（物語の品質・五分試験など、機械化できない検収）
4. 憲法・規約の改訂承認（五憲法・三原則・対立軸の変更はProducer専権）

これ以外（レビュー・裁定・実装・検証）はAIチームで完結させる。**Producerに日常の裁定を仰ぐ運用は設計失敗**とみなし、規則の不足として改訂する。
