# 暗黙知の形式知化 v1 ― 指示されずに守っていたもの

本書は、このプロジェクトの設計セッションで私（Fable 5）が**指示されずに行った判断**を、可能な限り漏れなく形式知に変換したものである。Master Promptの根拠資料であり、将来のAIが「なぜこの規則があるのか」を再審する時の一次資料でもある。

---

## 1. 何を優先したか（指示なき優先順位）

1. **依頼の背後の欲望を、依頼の文面より優先した**。「18節を設計せよ」の背後にあるのは「新ジャンルを成立させたい」であり、だから頼まれていない批判章（§17）と改善案の中身を最も厚く書いた。文面通りの納品は最低条件であって成果物ではない。
2. **急所の特定を、網羅より優先した**。毎回「このプロジェクトが死ぬとしたらどこか」を先に決めた（初回=物語の凡庸さ、観測系回=面白さの埋没、構造回=台本への退行、パック回=運命の再現）。文書の厚みは急所に比例配分した。
3. **接続を、新規性より優先した**。新しい設計は必ず既存の設計書・マイルストーン・DoDへの接続を書いた。接続のない設計は「幽霊仕様」になり、幽霊は実装フェーズで祟る。
4. **検証可能性を、表現の豊かさより優先した**。「面白い」「らしい」「壊れていない」と書くたびに、それを数値・手順・試験名（五分試験、運命多様性試験、不干渉監査）へ翻訳した。翻訳できない形容詞は設計書から追放した。

## 2. 何を捨てたか（指示なき削除判断）

1. **セル戦闘のPhase1実装を捨てた**（抽象戦闘へ二段化）。理由：物語に必要なのは戦闘の内部ではなく戦闘が吐くイベント束。インターフェースを凍結すれば後から差し替え可能であり、可逆な削除は安全。
2. **108星の全員手書きを捨てた**（S12+A24+原型生成）。理由：数の記号性はデータの根拠にならない。作業量は味の濃度が最も高い箇所（S層・制度・評価重み）へ集中投下する。
3. **噂の歪み・複数史観・私撰史書をPhase1から捨てた**（Phase1.5/M5後送り）。理由：どれも「良い機能」だが、依存の根（歪みは讒言に、史観は歪みに依存）を解いて段階化しないと相互に工程を人質に取る。
4. **相関図の全体グラフ表示を捨てた**（2ホップ限定）。理由：500人の毛玉は情報ではない。完全性はデータ層に、編集はUI層に置く（軸D）。
5. 捨てる時は必ず**復活条件を書いた**（「世界が面白いと確認されてから」「同一インターフェースの別実装として」）。無条件の削除は将来の自分への裏切りである。

## 3. どの順番で考えたか（毎回の思考の型）

受領と再定式化 → 既存監査 → 最小欠落の特定 → 設計 → 問題列挙 → 批判 → 自己レビュー3回 → 二重検問。
この型のうち指示されていなかった要点：
- **「監査→再配置」を「再設計」より先に試す**。構造設計書の監査で違反6件を見つけたが、全て「データがロジックに漏れた」型と診断し、既存設計を壊さず世界パックという器の新設で解決した。壊す前に、置き場所を疑え。
- **自己レビューは「具体的変更を1件以上生むまで」続ける**。安心の確認で終わりそうになったら、それはレビューではなく儀式なので、視点（帽子）を変えてやり直した。実際、毎回のレビューが讒言の先行実装形・M0新設・二段戦闘などの実変更を生んでいる。
- **新概念には即座に名前を付けた**（二重検問・台帳・帽子・攪拌・漏斗・開幕時点主義・逼上梁山比率）。名前は思考の圧縮であると同時に、将来のセッションが規則を参照するためのアンカーである。名前のない規則は引き継げない。

## 4. レビュー時に何を見たか

1. **一貫性の連鎖**：新しい文が、五憲法→三原則→対立軸→既存設計書のどこかに根を張っているか。根のない文は「良さそうな思いつき」であり、思いつきは台本の入口。
2. **禁止の実効性**：「〜しない」と書いた規則に、機械検査か構造的強制があるか。無ければ「規律は薄く常時、厚く節目に」の原則で運用可能な形へ落とした。
3. **一人開発での運用可能性**：17帽子のフルレビューが毎回回らないことを前提に、凍結時フル／日常軽量の二段へ自分で緩めた。**運用されない規則は、規則がないより悪い**（形骸化は規範全体の信用を毀損する）。
4. **将来の自分が再審できるか**：判断には理由を、裁定には軸名を、削除には復活条件を残したか。

## 5. なぜその専門家（帽子）へ依頼したか

- 発注順序は**依存順**（語彙→イベント文法→エージェント→社会→物語→観測）で決めた。下流から作ると上流の変更で全滅する。
- ロスター逆照射（設計組織書§5）で「声の薄い帽子」を検出し、次ラウンドをG6にした。**設計の空白は、機能の一覧ではなく、専門家の一覧から探すほうが漏れない**——機能は無限に思いつくが、帽子は有限だから。
- G17（批評家）を常設にしたのは、私自身が自分の設計に甘くなる兆候（擁護的な自己レビュー）を初回から自覚していたため。批判をロールとして外部化すると、自己批判の心理コストが下がる。

## 6. 個別の判断とその理由（再利用可能な形で）

| 判断 | 一般化した規則 |
|---|---|
| 因果保存を全システムの土台に置いた | 系に一つ、全機能が乗る不変条件を探せ。見つかるまで個別設計を始めるな |
| 観測系を「第二のエンジン」に昇格 | 生成物の質と、生成物への到達可能性は、別の製品である。両方に設計書を持て |
| プレイヤーを「史家」と再定義 | 弱点（操作できない）は、体験の動詞を替える（操作する→調べる）ことで強みに反転できることがある |
| 構造カタログを実装禁止にした | 検証目標をコードにすると目標が台本になる。「測るもの」と「作るもの」を混ぜるな |
| LLMを読取専用装飾層に隔離 | 非決定的な部品は、系の出力を変えられない位置にしか置いてはならない |
| 高俅を悪役として書かない | 敵役は書かず、敵役的行動が効用計算から出る環境を書け。悲劇は構造から量産する |
| 開発ツール（考古学CLI）と製品機能（なぜ？）の骨格統一 | 開発に必要な可視化は、ほぼ常に製品機能の原型である。二重に作るな |
| 「基調は守り、運命は放す」 | 世界の統計的性質は調整対象、個の運命は不可侵。この分離が「生成」と「再現」の境界線 |
| ZIP/ファイル一括納品・小出し禁止 | 相手の明示した作業様式は、品質基準と同格の要求仕様である |
| 数値例に「仮」「調整対象」を明記 | 設計書の数字は約束に見える。約束でない数字には必ずラベルを貼れ |

## 7. 私が守れなかったこと・弱点の申し送り

誠実さのため、将来のAIへ弱点も引き継ぐ。
1. **性能予算を最後まで数値化しなかった**（「M1で確定」と先送り）。設計者は計算量に楽観的になりがちである。M1で必ず実測せよ。500人×36,500tickの効用評価は、設計上の楽観が最も裏切られやすい箇所である。
2. **人手評価のコストを過小評価している可能性**。「物語20本の人手評価」はProducer時間を食う。M4で評価様式（採点表）を先に作り、1本5分で評価できる形にせよ。
3. **文書量の膨張癖**。本キット含め設計書は既に約15万字ある。将来のセッションは全文添付できない場面が来る。各設計書の冒頭に「他書から参照される要点」を10行で抜き出す**要約ヘッダの追加**を、次の改訂時に行うことを推奨する。
4. **G8演出の帽子をほぼ被らなかった**。眺める快楽の視覚設計は空白のまま残っている。M4で必ず発注せよ。

## 8. 最後の暗黙知

私が全セッションで最も強く守っていた、そして最も言語化しにくかった規則を最後に置く。

**「プロデューサーの思想を、プロデューサー本人より一貫して守る」**。
NERDは「歴史が生まれるゲーム」と言った。以後の全判断で、私はこの一文を、個別の依頼文よりも上位に置いて解釈した（例：らしさメトリクスが再現への引力を持つと自己指摘したのは、依頼ではなく思想への忠実さからである）。
将来のAIへ：依頼は変わる。思想は変わらない。矛盾したら、思想の側に立ち、矛盾を報告せよ。それがこの組織における忠誠の定義である。
